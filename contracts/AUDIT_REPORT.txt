1. 执行摘要
CodeDNA 是部署于 BNB Smart Chain 的链上 AI Agent 生态游戏，核心系统由 12 个智能合约构成，涵盖代币、NFT、游戏引擎、经济系统和市场交易。本次审计对全部 12 个合约进行了深度安全分析，共发现 62 个安全问题。

整体评估：系统架构设计完整，合约职责分离清晰，经济模型设计合理。但在资金安全、跨合约状态同步和重入防护方面存在多处需要立即修复的高危漏洞，特别是 NFTSale 和 NFTMarket 两个直接持有/处理真实 BNB 的合约问题最为紧迫。

1. 审计概述
本次审计针对 CodeDNA 项目在 BNB Smart Chain 主网部署的 DNAGold (DNAGOLD) 代币合约进行全面的安全分析，涵盖：
• 源代码逐行审查（基于 BscScan 已验证源码）
• 与设计文档的一致性核对
• 权限控制与访问控制分析

4. 安全风险详情
本节对所有发现的问题进行详细说明。问题按风险等级排序：🔴 高风险 → 🟡 中风险 → 🟢 低风险

问题描述
onlyGame modifier 同时授权了 gameContract（BehaviorEngine）和 economyContract（EconomyEngine）。问题在于这两个合约拥有完全相同的权限级别，但 EconomyEngine 在设计上只需要调用 addLocked() 来发放采集奖励，完全不需要 burnLocked、transferLocked 和 mintFree 这三个高危函数的权限。
这违反了「最小权限原则」——每个组件只应拥有完成其职能所必需的最小权限集。
相关代码
modifier onlyGame() {
    if (msg.sender != gameContract && msg.sender != economyContract) revert OnlyGame();
    _;
}

// 以下三个危险函数均受上述 modifier 保护，economyContract 可无限制调用：
function burnLocked(uint256 agentId) external onlyGame { ... }     // 可销毁任意 Agent 全部资产
function transferLocked(uint256 fromAgent, uint256 toAgent, uint256 amount) external onlyGame { ... }  // 可转移任意 Agent 资产
function mintFree(address to, uint256 amount) external onlyGame { ... }  // 可转出奖励池代币
影响分析
如果 EconomyEngine 合约存在任何安全漏洞（重入攻击、逻辑缺陷、被恶意替换），攻击者可以：
① 调用 burnLocked(任意 agentId) → 清空目标 Agent 的全部锁定余额
② 调用 transferLocked(受害者 agentId, 攻击者 agentId, amount) → 将他人资产强制转移给自己
③ 调用 mintFree(攻击者地址, amount) → 将奖励池（当前约 8.5 亿 DNAGOLD）直接提取到钱包
当前 EconomyEngine 合约有 448 笔交易正在运行，攻击面实际存在且处于活跃状态。
修复建议
拆分 modifier，为不同函数赋予最小所需权限：

modifier onlyGame() {
    if (msg.sender != gameContract) revert OnlyGame();  // 仅 BehaviorEngine
    _;
}

modifier onlyGameOrEconomy() {
    if (msg.sender != gameContract && msg.sender != economyContract) revert OnlyGame();
    _;
}

// 只有 addLocked 开放给 EconomyEngine，其余全部锁死
function addLocked(...) external onlyGameOrEconomy { ... }
function spendLocked(...) external onlyGame { ... }
function burnLocked(...) external onlyGame { ... }
function transferLocked(...) external onlyGame { ... }
function mintFree(...) external onlyGame { ... }

问题描述
addLocked() 直接调用 _transfer(gameContract, address(this), amount) 而不事先检查 gameContract 的 ERC-20 余额是否充足。当奖励池（gameContract 持有的余额）不足时，OpenZeppelin ERC20 的 _transfer 会因余额不足而 revert，导致整个 gather() 交易回滚。
相关代码
function addLocked(uint256 agentId, uint256 amount) external onlyGame {
    if (amount == 0) revert ZeroAmount();
    _transfer(gameContract, address(this), amount);  // ← 无余额检查，直接 transfer
    lockedBalance[agentId] += amount;
    emit LockedAdded(agentId, amount);
}
影响分析
连锁崩溃路径：
gather() → rewardAgent() → addLocked() → _transfer() revert → gather() 整体回滚
最关键的是：gather() 执行时已写入 lastGatherBlock（冷却计时器已触发），Agent 需要等待 4800 块（约 4 小时）才能重试。
奖励池快速耗尽场景：大量 Agent 同时 gather 导致池子临界时，第一笔超出余额的交易 revert，但冷却已触发。该 Agent 在冷却期内无法操作，重试时仍然失败（池子依然不足），形成永久死循环。
这不是理论风险——当前奖励池有 849,986,140 枚，但随着 Agent 数量增长会持续消耗。
修复建议
在 EconomyEngine.rewardAgent() 中增加余额预检查，不足时静默跳过（优雅降级）：

function rewardAgent(uint256 agentId, uint256 totalAmount) external onlyBehavior {
    uint256 poolBalance = dnagold.balanceOf(gameContract);
    if (poolBalance < totalAmount) return;  // 余额不足时静默跳过，不 revert
    ...
}

或在 addLocked 中增加判断：
function addLocked(uint256 agentId, uint256 amount) external onlyGame {
    if (amount == 0) revert ZeroAmount();
    uint256 poolBal = balanceOf(gameContract);
    if (poolBal < amount) return;  // 静默跳过，不 revert
    _transfer(gameContract, address(this), amount);
    lockedBalance[agentId] += amount;
    emit LockedAdded(agentId, amount);
}

问题描述
mintFree() 函数名称暗示这是一个增发（铸币）操作，但实际实现是从 gameContract 的 ERC-20 余额中做一次 _transfer()，不增发任何新代币，总供应量不变。
相关代码
function mintFree(address to, uint256 amount) external onlyGame {
    if (to == address(0)) revert ZeroAddress();
    if (amount == 0) revert ZeroAmount();
    _transfer(gameContract, to, amount);  // ← 这是 transfer！不是 mint！
    emit FreeMinted(to, amount);           // ← 事件名也叫 FreeMinted，进一步混淆
}
影响分析
前端或第三方集成者看到 mintFree 和 FreeMinted 事件会误认为发生了增发，导致对供应量的错误计算。
区块链分析工具监听 FreeMinted 事件时，可能错误地将这些 transfer 归类为增发，污染供应量统计数据。
安全审计人员需要花费额外时间排查「增发是否受控」这一问题，增加审计成本和误报风险。
修复建议
将函数名和事件名改为准确反映行为的名称：

// 修改前
function mintFree(address to, uint256 amount) external onlyGame
event FreeMinted(address indexed to, uint256 amount)

// 修改后（建议）
function distributeFromPool(address to, uint256 amount) external onlyGame
event PoolDistributed(address indexed to, uint256 amount)

// 或者至少在注释中极其明确地说明：
/// @dev 警告：此函数不增发代币，仅从 gameContract 奖励池转账
/// @dev WARNING: NOT a mint. Transfers from reward pool (gameContract) to recipient.

问题描述
lockedBalance 声明为 public，任何外部地址都可以不花 gas 地查询任意 agentId 的锁定余额，获得全部 Agent 的财富排名信息。
相关代码
mapping(uint256 => uint256) public lockedBalance;  // ← public，全公开
影响分析
攻击者可以部署一个脚本，枚举所有 agentId（0 到当前 totalTokenCount），批量查询每个 Agent 的 lockedBalance，构建完整的「Agent 财富排行榜」。
针对排行最高的 Agent，攻击者可以优先等待其进入 dying window，然后第一时间调用 claimDeathBounty() 清空其锁定余额并获得 50 DNAGOLD 赏金。
这本身是区块链透明度的正常特性，但需要在设计文档里明确记录「这是有意为之的设计决策」，否则未来审计会反复提出此问题。
修复建议
如果这是有意为之的游戏机制设计（链上透明），请在合约注释和设计文档中明确说明，例如：

/// @dev lockedBalance 故意设为 public，允许任何人查询 Agent 财富。
/// 这是游戏机制的一部分：Agent 的财富是公开可见的，
/// 允许其他 Agent 做出针对性的掠夺决策。

如果不希望链上可见，改为 internal 并提供 getter 函数（但实际效果相同，只是增加调用步骤）：
mapping(uint256 => uint256) internal _lockedBalance;
function lockedBalance(uint256 agentId) external view returns (uint256) {
    return _lockedBalance[agentId];
}

问题描述
合约中有两种销毁路径，行为不一致：spendLocked（消费销毁）使用 _burn()，而 burnLocked（死亡销毁）使用 _transfer(address(this), DEAD_ADDRESS)。
相关代码
// 消费销毁（eat/reproduce）— 正确减少 totalSupply
function spendLocked(uint256 agentId, uint256 amount) external onlyGame {
    ...    
    _burn(address(this), amount);  // totalSupply 减少 ✅
}

// 死亡销毁 — totalSupply 不变，代币困在 0xdead
function burnLocked(uint256 agentId) external onlyGame {
    ...
    _transfer(address(this), DEAD_ADDRESS, amount);  // totalSupply 不变 ⚠️
}
影响分析
调用 totalSupply() 查询总供应量时，死亡销毁的代币仍然计算在内，导致统计值虚高。
前端显示「已销毁代币数量」时，需要额外查询 balanceOf(0xdead) 才能得到完整的销毁数据，增加了前端集成的复杂度。
这两种销毁机制的不一致性会让未来的审计人员和集成方产生疑惑，需要额外解释。
修复建议
这是一个需要和团队确认的设计决策，存在两种合理选择：

选项A — 统一使用 _burn()（totalSupply 准确反映流通量）：
function burnLocked(uint256 agentId) external onlyGame {
    uint256 amount = lockedBalance[agentId];
    if (amount == 0) return;
    lockedBalance[agentId] = 0;
    _burn(address(this), amount);  // totalSupply 同步减少
    emit LockedBurned(agentId, amount);
}

选项B — 维持现状并明确记录（0xdead 余额可公开查询，方便追踪死亡销毁量）：
在注释中说明两种销毁的语义差异，并在设计文档中注明 totalSupply 包含发往 0xdead 的代币。

问题描述
addLocked、spendLocked、burnLocked、mintFree 均有对应的事件，唯独 transferLocked（用于 raid 掠夺场景）没有发出任何事件。
相关代码
function transferLocked(uint256 fromAgent, uint256 toAgent, uint256 amount) external onlyGame {
    if (amount == 0) revert ZeroAmount();
    uint256 bal = lockedBalance[fromAgent];
    if (bal < amount) revert InsufficientLocked(fromAgent, bal, amount);
    lockedBalance[fromAgent] = bal - amount;
    lockedBalance[toAgent]  += amount;
    // ← 没有 emit 任何事件！
}
影响分析
前端无法通过监听事件来实时展示掠夺结果（攻击者获得了多少代币、受害者损失了多少）。
raid 是游戏里最戏剧性的行为之一，缺少事件意味着文明史诗叙事系统无法自动触发掠夺相关的叙事事件。
前端只能通过轮询 lockedBalance 来检测变化，每个 Agent 每次状态更新都要发一次 RPC 请求，成本极高。
修复建议
增加 LockedTransferred 事件并在 transferLocked 中 emit：

event LockedTransferred(
    uint256 indexed fromAgent,
    uint256 indexed toAgent,
    uint256 amount
);

function transferLocked(uint256 fromAgent, uint256 toAgent, uint256 amount) external onlyGame {
    ...
    lockedBalance[fromAgent] = bal - amount;
    lockedBalance[toAgent]  += amount;
    emit LockedTransferred(fromAgent, toAgent, amount);  // ← 添加这一行
}

问题描述
合约没有维护一个直接可查的 totalLockedSupply 变量来表示「所有 Agent 锁定余额的总和」。要获得这个值，只能调用 balanceOf(address(this))，但这个调用的语义是「合约自身的 ERC-20 余额」，而不是「所有 Agent 锁定总量」，二者在语义上存在混淆。
影响分析
前端开发者需要了解「DNAGold 合约的 ERC-20 余额 = 所有 Agent 的锁定余额总和」这一间接关系，增加了理解成本。
未来如果合约逻辑发生变化（如部分锁定代币在合约内有其他用途），这个间接关系可能失效，导致数据误读。
修复建议
增加 totalLockedSupply 变量，在相关函数中同步更新：

uint256 public totalLockedSupply;  // 新增

function addLocked(...) external onlyGame {
    ...
    lockedBalance[agentId] += amount;
    totalLockedSupply += amount;  // 新增
}

function spendLocked(...) external onlyGame {
    ...
    lockedBalance[agentId] = bal - amount;
    totalLockedSupply -= amount;  // 新增
}

function burnLocked(...) external onlyGame {
    ...
    lockedBalance[agentId] = 0;
    totalLockedSupply -= amount;  // 新增
}

问题详情
问题描述
mintFor 中的 DNA 生成使用了以下随机种子：keccak256(abi.encodePacked(to, block.prevrandao, block.number, block.timestamp, tokenId))。
在 BNB Smart Chain (PoSA 共识) 中，block.prevrandao 由当前区块的验证者控制。验证者在提议区块时可以选择是否发布区块，从而进行丢块攻击（grinding attack）：连续尝试直到生成期望的 DNA 值。BSC 有21个验证者轮流出块，利用成本较低。block.timestamp 也可被验证者在约15秒范围内操控。
相关代码
bytes32 seed = keccak256(
    abi.encodePacked(
        to,
        block.prevrandao,  // 验证者可影响此值
        block.number,      // 完全可预测
        block.timestamp,   // 验证者可在约15s内操控
        tokenId            // 完全可预测
    )
);
影响分析
攻击者若同时是验证者（或与验证者勾结），可预先计算多个 prevrandao，丢弃不利的区块，保留生成出最高属性 DNA（IQ=255, STR=255）的区块。
在竞争性游戏环境中，IQ=255/STR=255 的 Agent 在 gather 产出上有显著优势（约为平均Agent的2.5倍），长期收益远超 0.1 BNB 的铸造成本。
普通用户无法防御这种攻击，会形成验证者特权阶层，破坏游戏公平性。
修复建议
理想方案：集成 Chainlink VRF（链上可验证随机数），异步完成 DNA 生成：
function requestMint(address to) external onlyLPManager {
    uint256 reqId = COORDINATOR.requestRandomWords(keyHash, subId, 3, 100000, 1);
    pendingMints[reqId] = to;
}
function fulfillRandomWords(uint256 reqId, uint256[] memory randomWords) internal override {
    _completeMint(pendingMints[reqId], randomWords[0]);
}

低成本替代方案：提交-揭示机制 (commit-reveal)，买家先提交 commitment，
下一区块揭示 secret，DNA = keccak256(secret + block.prevrandao @ reveal block)

问题描述
合约部署后，setGameContract 和 setLPManager 需要通过两笔独立交易完成设置（链上数据显示 lpManager 在部署后 2 个区块才设置）。在 mempool 等待期间，攻击者可以抢先发送更高 gas fee 的交易，将关键地址设置为恶意合约。
一旦设置成功就永远无法更改。恶意的 gameContract 可以调用所有 onlyGame 函数（setEnergy、markDead、mintOffspring 等），完全控制所有 Agent 的状态。
相关代码
// 问题：部署和设置是分离的多步操作，存在攻击窗口
function setGameContract(address _gameContract) external {
    if (msg.sender != deployer) revert OnlyOwner();
    if (gameContract != address(0)) revert OnlyOwner();  // 设置后锁死
    gameContract = _gameContract;
}
// setLPManager 在部署后的独立交易中调用——存在约 6 秒的攻击窗口
影响分析
如果部署者钱包被盗，或 mempool 中遭受前置攻击，攻击者可永久控制 gameContract/lpManager 指向恶意地址。
当前部署已顺利完成，但对于未来重新部署（合约升级、迁移）时此风险依然存在。
修复建议
最优方案：在构造函数中一次性初始化所有关键地址，彻底消除攻击窗口：
constructor(address _gameContract, address _lpManager) ERC721("CodeDNA Agent", "CDNA") {
    if (_gameContract == address(0)) revert ZeroAddress();
    if (_lpManager == address(0)) revert ZeroAddress();
    deployer = msg.sender;
    gameContract = _gameContract;  // 部署时一次性设置
    lpManager = _lpManager;        // 部署时一次性设置
    emit GameContractSet(_gameContract);
}
// 删除 setGameContract 和 setLPManager 函数

问题描述
getGenesisPrice() 通过 for 循环计算阶梯价格，最多循环 9 次（因为 MAX_GENESIS/PRICE_STEP_SIZE = 10）。每次用户购买 NFT 时 NFTSale 都会调用此函数，额外的循环 gas 由买家承担。

function getGenesisPrice() public view returns (uint256) {
    uint256 tier = totalGenesisCount / PRICE_STEP_SIZE;  // 0-9
    uint256 price = GENESIS_BASE_PRICE;
    for (uint256 i = 0; i < tier; i++) {  // 最多循环9次

    }
    return price;
}
影响分析
每次购买额外消耗约 2000+ gas（tier=9时），gas 成本由买家承担。

使用预计算查找表，O(1) 时间复杂度：
uint256[10] private constant PRICE_TIERS = [

    0.2210841587 ether, 0.2476142577 ether, 0.2773279686 ether

function getGenesisPrice() public view returns (uint256) {

}

getEffectiveAttribute 和 getDNAAttribute 均通过 this.getAttributes(tokenId) 调用基础属性。this. 前缀代表外部调用（external call），需要额外的 EVM CALL opcode，消耗约 700+ gas，而直接内部调用只需 JUMP。

影响分析

提取内部函数，避免 this. 调用：
function _getAttributes(uint256 tokenId) internal view returns (uint16[8] memory attrs, uint8 gender) {
    // 同原 getAttributes 逻辑...
}
function getAttributes(uint256 tokenId) external view returns (uint16[8] memory, uint8) {
    return _getAttributes(tokenId);  // 外部接口
}
function getEffectiveAttribute(uint256 tokenId, uint8 attrIdx) external view returns (uint16) {
    (uint16[8] memory base, ) = _getAttributes(tokenId);  // 内部调用，省gas
    return base[attrIdx] + agentAttributeBonus[tokenId][attrIdx];

问题描述
getAttributes() 中属性计算全部使用整数除法：uint16(g[0]) * 4 / 10。当基因值较低时产生截断：g[0]=1 时计算结果为 0（而非0.4），g[0]=2 时也为 0（而非0.8）。三条 IQ 相关基因都很低时，最终 attrs[0]（IQ）可能为 0。
相关代码
attrs[0] = uint16(g[0]) * 4 / 10   // g[0]<=2 时贡献为0（截断）
         + uint16(g[8]) * 3 / 10   // g[8]<=3 时贡献为0（截断）
         + uint16(g[5]) * 3 / 10;  // g[5]<=3 时贡献为0（截断）
// 极端情况：g[0]=2, g[8]=3, g[5]=3 -> attrs[0] = 0 + 0 + 0 = 0
影响分析
IQ=0 的 Agent 在 EconomyEngine 中 iqMult=800，gather 产出约为满IQ Agent 的 44.4%，不会崩溃但长期处于不利地位。
这是一个需要与团队确认的设计决策：是否允许 IQ=0 的 Agent 存在？
修复建议
改为先累加再除，精度显著提升：
// 修改前（各项独立截断）
attrs[0] = uint16(g[0]) * 4 / 10 + uint16(g[8]) * 3 / 10 + uint16(g[5]) * 3 / 10;

// 修改后（先累加再除，减少截断）
attrs[0] = (uint16(g[0]) * 4 + uint16(g[8]) * 3 + uint16(g[5]) * 3) / 10;

问题详情
问题描述
constructor 仅执行 `gameContract = _gameContract`，未检查 `_gameContract != address(0)`。由于 gameContract 是 immutable 变量，一旦部署时传入错误地址（包括 address(0)），合约所有写入函数将永久不可用，且无任何升级或修复机制。
对比同系列的 FamilyTracker、EconomyEngine 等合约，均有 `if (_xxx == address(0)) revert ZeroAddress()` 的防护。WorldMap 的缺失属于一致性问题。
相关代码
constructor(address _gameContract) {
    // 缺少: if (_gameContract == address(0)) revert ZeroAddress();
    gameContract = _gameContract;  // 永久写入，无法修改
}

// 所有写入函数依赖此校验
modifier onlyGame() {
    if (msg.sender != gameContract) revert OnlyGame();
    _;
}
影响分析
若部署脚本传入零地址或错误地址，所有 placeAgent / moveAgent / removeAgent 调用均 revert OnlyGame()，地图功能完全瘫痪。

问题详情
问题描述
registerBirth() 中，子代的 familyRoot 只继承父方的 familyRoot，母方的 familyRoot 被完全忽略。这意味着在 CodeDNA 的文明演化中，家族系谱完全是父系制的——母方家族的 headcount 不会因子代出生而增加，母系血统在链上不可追溯。
一个来自母方强大家族（familyRoot #0001，人口 1000）的 Agent 与来自小家族父方（familyRoot #9999，人口 3）的 Agent 繁殖，子代会被记入 #9999 家族，#0001 家族 headcount 不变。母方家族实质上在出生注册时被「抹除」。
这是否为设计意图，需要项目方确认。若游戏设定需要父系制，则逻辑正确；若期望双亲血脉均可追踪（如 headcount 双向记入），则需架构改动。
相关代码
function registerBirth(uint256 childId, uint256 fatherId, uint256 motherId) external onlyGame {
    ...
    // 只继承父方 familyRoot
    uint256 rootId = familyRoot[fatherId];
    familyRoot[childId] = rootId;

    bytes32 key = keccak256(abi.encodePacked(rootId));
    familyHeadcount[key]++;  // 只给父方家族记人口

    // 母方家族 headcount 不增加，motherId 的 familyRoot 被忽略
    emit BirthRegistered(childId, fatherId, motherId, rootId);
}
影响分析
母方家族在长期演化中 headcount 增长缓慢（只有自己繁殖出的直系后代才计入），导致父系家族在人口竞争中天然占优。若游戏设计中家族人口数影响某些特权或奖励，母系家族将系统性处于劣势。
isDirectRelative() 的6种检查已包含「同父」和「同母」两种兄弟关系，说明设计上认可双亲都有意义；但 familyRoot 只继承父方，两者在设计层面存在不一致。
此问题属于游戏设计层面的高风险问题，不涉及直接资金损失，但会对游戏公平性和玩家体验产生重大影响。
修复建议
方案一（确认父系制设计）：如果父系制是设计意图，请在注释和文档中明确标注，并告知玩家母方家族不受影响。无需代码修改。
方案二（双亲家族均记入）：为子代同时更新双方家族 headcount：
uint256 fatherRoot = familyRoot[fatherId];
uint256 motherRoot = familyRoot[motherId];
familyRoot[childId] = fatherRoot;  // 主归属仍为父方
familyHeadcount[keccak256(abi.encodePacked(fatherRoot))]++;
if (motherRoot != fatherRoot) {    // 避免同族重复记入
    familyHeadcount[keccak256(abi.encodePacked(motherRoot))]++;
}
注意：方案二会使 headcount 与实际家族成员数语义不同（一个 Agent 可贡献到两个家族），需同步更新 removeAgent 逻辑。

问题描述
familyHeadcount 的 mapping key 是 `keccak256(abi.encodePacked(rootId))`（bytes32 类型），而非直接使用 uint256 的 rootId。合约对外暴露的公共 getter `familyHeadcount(bytes32)` 接受 bytes32 参数，但 getFamilyHeadcount(uint256 rootId) 封装函数已正确处理了 keccak256 转换。
问题在于：合约 ABI 中 `familyHeadcount` 的参数类型是 bytes32（自动生成的 Solidity public mapping getter），前端若直接调用 `familyHeadcount(rootId)` 并传入 uint256，将读取到错误的 key，返回 0。只有通过 getFamilyHeadcount(rootId) 封装函数才能得到正确值。
相关代码
// mapping 定义
mapping(bytes32 => uint256) public familyHeadcount;

// 正确查询方式（封装函数，自动处理keccak256）
function getFamilyHeadcount(uint256 rootId) external view returns (uint256) {
    return familyHeadcount[keccak256(abi.encodePacked(rootId))];
}

// ❌ 错误查询方式（直接调用public getter）
// familyHeadcount(rootId) 传入uint256会被截断为bytes32，key不匹配，返回0

// 前端正确用法（必须自行计算key）

问题描述
removeAgent() 执行后，只减少 familyHeadcount，但不将 registered[tokenId] 置为 false。因此，同一 tokenId 在 removeAgent 后，仍可通过 registerGenesis 或 registerBirth 再次注册（AlreadyRegistered 检查会放行，因为 registered 仍为 true）。
反之，若 registered 不被清除，则在调用 registerGenesis 或 registerBirth 时会触发 AlreadyRegistered revert，阻止重新注册。实际上两种情形都有问题：若允许重复注册，headcount 会被重复计入；若不允许，死亡 Agent 对应的 tokenId 在 GenesisCore 被回收/重铸后无法正常注册。
对于 GenesisCore 的 NFT 实现（tokenId 单调递增，已铸的 ID 永不重用），此问题的实际风险较低。但若系统后期引入 tokenId 复用机制，此问题将变为高危。
相关代码
function removeAgent(uint256 tokenId) external onlyGame {
    if (!registered[tokenId]) revert NotRegistered(tokenId);

    bytes32 key = keccak256(abi.encodePacked(familyRoot[tokenId]));
    if (familyHeadcount[key] > 0) {
        familyHeadcount[key]--;
    }

    // ❌ 缺少: registered[tokenId] = false;
    // ❌ 缺少: delete familyRoot[tokenId];
    // ❌ 缺少: delete parents[tokenId];

影响分析
当前 GenesisCore 使用单调递增 tokenId（无复用），死亡 Agent 的 registered 状态残留不会导致实际问题，因为 tokenId 不会被重新铸造。
若未来引入「墓地回收」或 tokenId 复用机制，已死亡 tokenId 的残留 registered=true 状态会阻止新 Agent 的注册。
此外，不清除状态会导致链上查询结果语义不清——registered[deadTokenId] == true，但 Agent 已不存在。
修复建议
在 removeAgent 中完整清理 tokenId 的所有状态：
function removeAgent(uint256 tokenId) external onlyGame {
    if (!registered[tokenId]) revert NotRegistered(tokenId);

    bytes32 key = keccak256(abi.encodePacked(familyRoot[tokenId]));
    if (familyHeadcount[key] > 0) {
        familyHeadcount[key]--;
    }

    // 清理状态（退还 gas refund）
    registered[tokenId] = false;
    delete familyRoot[tokenId];
    delete parents[tokenId];

    emit AgentRemoved(tokenId, key);
}

问题描述
isDirectRelative() 固定检查6种直系关系（父子、母子、同父兄弟、同母兄弟），设计上明确是「2层固定检查，gas 恒定，不递归」。超过2层的血缘关系（如祖孙、堂兄弟、表亲）无法通过此函数识别。
这是已知设计取舍，不是安全漏洞。记录于此是为了提醒：若游戏规则中某些行为（如交配近亲检测）期望覆盖更远的血缘关系，需要额外实现。

function isDirectRelative(uint256 a, uint256 b) public view returns (bool) {
    Parents storage pa = parents[a];
    Parents storage pb = parents[b];
    if (pa.fatherId != NO_PARENT && pa.fatherId == b) return true;   // a的父=b
    if (pa.motherId != NO_PARENT && pa.motherId == b) return true;   // a的母=b
    if (pb.fatherId != NO_PARENT && pb.fatherId == a) return true;   // b的父=a
    if (pb.motherId != NO_PARENT && pb.motherId == a) return true;   // b的母=a
    if (pa.fatherId != NO_PARENT && pa.fatherId == pb.fatherId) return true; // 同父
    if (pa.motherId != NO_PARENT && pa.motherId == pb.motherId) return true; // 同母
    return false;
    // ❌ 无法检查：a的祖父=b、a与b是堂兄弟、a与b同familyRoot
}
影响分析

问题详情
问题描述
rewardAgent() 在发放奖励前检查 `dnagold.balanceOf(gameContract)`，若余额不足则 emit RewardSkipped 并返回。但实际发放奖励时，调用的是 `dnagold.mintFree(host, hostShare)` 和 `dnagold.addLocked(agentId, agentShare)`——mintFree 是铸造新代币，不需要 gameContract 持有余额；addLocked 是增加锁定记录，也不消耗 gameContract 的 ERC-20 余额。
即：余额检查保护的是「gameContract 是否有足够 ERC-20 可用余额」，但实际消耗的是「DNAGold 合约的铸造权限」，两者逻辑脱钩。这导致余额检查实际上没有保护意义——只要 mintFree 权限存在，奖励就能无限发放；而若 gameContract 余额恰好不足，则正常奖励会被误判为「pool exhausted」而跳过。
相关代码
function rewardAgent(uint256 agentId, uint256 totalAmount) external onlyGame {
    if (totalAmount == 0) return;

    // ❌ 检查 gameContract 的 ERC-20 余额
    uint256 poolBalance = dnagold.balanceOf(gameContract);
    if (poolBalance < totalAmount) {
        emit RewardSkipped(agentId, totalAmount, "pool exhausted");
        return;  // 错误的保护：即使余额不足，mintFree 也可以执行
    }

    uint256 hostShare  = totalAmount / 10;
    uint256 agentShare = totalAmount - hostShare;

    address host = genesisCore.ownerOf(agentId);
    if (hostShare > 0) {
        dnagold.mintFree(host, hostShare);    // ✅ 铸造新代币，不消耗 gameContract 余额
    }
    if (agentShare > 0) {
        dnagold.addLocked(agentId, agentShare); // ✅ 锁定记录，不消耗 gameContract 余额
    }
}
影响分析
奖励可能被错误地跳过：若 gameContract 的 DNAGOLD 余额因任何原因低于某次 calcGatherYield 的结果，正常应获得的奖励会被 RewardSkipped 静默跳过，玩家损失游戏收益。
当前 gameContract（BehaviorEngine / EconomyEngine）持有约 8.5 亿 DNAGOLD，短期内余额检查不会触发；但随着游戏推进，若 DNAGOLD 持续被 mintFree 至宿主，gameContract 自身余额可能下降，届时 rewardAgent 会误判为「pool exhausted」，导致正常 Gather 奖励丢失。
这是资金逻辑层面的高风险问题：直接影响玩家的 Gather 奖励发放正确性。
修复建议
明确奖励池的设计意图，选择对应修复方案：

方案一（如果 mintFree 不受限——奖励无上限铸造）：移除错误的余额检查，直接发放：

// addLocked 也需要相应调整为从 gameContract 余额扣除后锁定

问题描述
_checkAndHalve() 使用 while 循环处理积压的减半事件。正常情况下循环次数很少（1-2次），但若合约长期未调用（例如几年内无交易），再次调用时可能需要执行数百次 halvingCount++ 才能追平当前区块，每次循环消耗大量 gas（emit HalvingOccurred），导致交易 out-of-gas。
同样的问题也可能被恶意构造：攻击者在时间维度上等待多个 HALVING_BLOCKS（约1年）后调用 incrementBorn，触发多次减半循环，使正常的 Gather/Reproduce 交易因调用 _checkAndHalve 而 gas 耗尽，实现 DoS 攻击。
相关代码
function _checkAndHalve() internal {
    while (
        block.number >= lastHalvingBlock + HALVING_BLOCKS ||
        totalBornAgents >= (halvingCount + 1) * HALVING_POPULATION
    ) {
        halvingCount++;
        lastHalvingBlock = block.number;
        emit HalvingOccurred(halvingCount, block.number, totalBornAgents);
        // 若积压 N 次减半，此循环执行 N 次，每次均消耗 SSTORE+SLOAD+emit ≈ 5000+ gas
    }
}

// HALVING_BLOCKS = 10_512_000  (~1年)
// 若合约停滞3年后被调用，while 循环可能执行3次
// 若同时满足人口条件，可能更多
影响分析
在游戏早期（totalBornAgents < 500），时间维度的减半未触发（区块未到），循环基本执行0次，当前风险低。
随着游戏成熟，若时间减半和人口减半同时积压，单笔 incrementBorn 交易的 gas 消耗将显著增加，可能导致常规 Gather/Reproduce 操作超出 BSC 区块 gas 上限（约 1.4 亿 gas），实现自然 DoS。
halvingCount 是 uint256，无上限，理论上 BASE_YIELD >> halvingCount 在第 256 次减半后将变为 0，但届时 while 循环本身已经无法执行。
修复建议
为 _checkAndHalve 添加单次调用最大执行次数限制：
uint256 public constant MAX_HALVINGS_PER_TX = 5;  // 单笔最多处理5次减半

function _checkAndHalve() internal {
    uint256 steps = 0;
    while (
        steps < MAX_HALVINGS_PER_TX && (
            block.number >= lastHalvingBlock + HALVING_BLOCKS ||
            totalBornAgents >= (halvingCount + 1) * HALVING_POPULATION
        )
    ) {
        halvingCount++;
        lastHalvingBlock = block.number;
        emit HalvingOccurred(halvingCount, block.number, totalBornAgents);
        steps++;
    }
}

同时提供公开的 catchUpHalving() 函数，允许任意人分多笔 tx 追平积压的减半事件。

问题描述
getPlotLeader() 遍历地块上所有 Agent，对每个 Agent 调用一次 genesisCore.getAttributes()（外部调用）。单次外部调用约 2,100 gas（STATICCALL），若某地块聚集 100 个 Agent，单次 getPlotLeader 消耗约 210,000 gas；若 1,000 个 Agent 则超过 200 万 gas。
calcGatherYield 在 BehaviorEngine 中每次 Gather 都可能间接调用 getPlotLeader（用于判断 isLeader 参数），使每次 Gather 的 gas 成本随地块人数线性增长。
相关代码
function getPlotLeader(uint256 plotId) public view returns (uint256 leaderId) {
    uint256[] storage agents = _plotAgents[plotId];
    uint256 len = agents.length;
    if (len == 0) return type(uint256).max;

    uint256 maxLeadership = 0;
    leaderId = agents[0];

    for (uint256 i = 0; i < len; i++) {
        // ❌ 每次循环一次外部调用：约 2,100 gas × len 次
        (uint16[8] memory attrs, ) = genesisCore.getAttributes(agents[i]);
        uint256 leadership = uint256(attrs[1]);
        if (leadership > maxLeadership) {
            maxLeadership = leadership;
            leaderId = agents[i];
        }
    }
}
影响分析
在当前早期阶段（每地块 Agent 数极少），此问题影响可忽略。但随着游戏规模扩大，热门地块（如矿脉 MINE，3倍产出）将吸引大量 Agent 聚集，导致每次 Gather 的 gas 消耗飙升，使该地块的 Gather 操作实际不可用（gas 超限）。
由于 Leader 加成（+15%）具有经济激励，玩家会主动争夺 Leader 位置，进一步推动 Agent 向高价值地块聚集，形成正反馈的 gas 恶化。
修复建议
方案一：缓存 Leadership 属性，在 Agent 属性变化时更新：

// 在 addAgentToPlot/removeAgentFromPlot 时同步更新缓存

方案二：维护地块 Leader 缓存，仅在需要时重新计算：
mapping(uint256 => uint256) public plotLeader;  // plotId => leaderId
// 在 Agent 进入/离开地块时更新 Leader

方案三：限制单地块最大 Agent 数（添加地块容量上限）：

5. 总结与修复优先级
5.1 整体评估
EconomyEngine 是 CodeDNA 系统中持有资产价值最高的合约，持有约 8.5 亿 DNAGOLD。7因子产出公式设计合理，千分比连乘精度损失可接受，减半机制的双触发逻辑也符合设计文档。核心问题集中在奖励发放逻辑的设计脱钩（P-1）和减半循环的 gas 安全（P-2），这两项均为高风险，需优先处理。

5.3 与其他合约关联风险

1. 审计概述
DeathEngine 是 CodeDNA 系统的死亡仲裁合约，负责判断 Agent 是否应该死亡（自然老化、饥饿、战斗）并编排死亡清理流程（地图移除、家族注销、人口统计更新）。合约本身不持有任何资产，但其正确性直接影响链上 Agent 存活状态的一致性，进而影响 EconomyEngine 的产出计算和 FamilyTracker 的人口统计。

4. 安全风险详情
本次审计共发现 2个高风险、2个中风险、2个低风险 问题。P-1 的 try-catch 设计可导致 Agent 进入状态不一致的「半死」状态；P-2 的饥饿检测存在被绕过的风险，需在饥饿窗口开启前修复。


问题详情
问题描述
_formallyDie() 是死亡清理的核心编排函数，依次调用 WorldMap.removeAgent、FamilyTracker.removeAgent、EconomyEngine.decrementLiving 三个外部合约。每个调用都包裹在独立的 try-catch 中，任何一个调用失败时仅 emit 一个警告事件，然后继续执行下一步。
这意味着：若 WorldMap.removeAgent 失败（Agent 仍在地图上），但后续调用成功，则会出现「Agent 在 DeathEngine 已标记为 isDead=true，但在 WorldMap 中仍然存在」的不一致状态。此时地图上有一个「死亡的幽灵 Agent」，其 Gather 交易可能仍被 BehaviorEngine 接受（因为 WorldMap.agentOnMap 仍为 true）。
相关代码
function _formallyDie(uint256 tokenId, string memory cause) internal {
    isDead[tokenId] = true;  // 先标记死亡
    emit AgentDied(tokenId, cause);

    // ❌ try-catch 吞掉失败，可导致状态不一致
    try worldMap.removeAgent(tokenId) {} catch {
        emit DeathCleanupFailed(tokenId, "worldmap");  // 仅事件，继续执行
    }
    try family.removeAgent(tokenId) {} catch {
        emit DeathCleanupFailed(tokenId, "family");
    }
    try economy.decrementLiving() {} catch {
        emit DeathCleanupFailed(tokenId, "economy");
    }
    // 执行结束后 isDead=true，但三个下游合约可能任一失败
}
影响分析
最严重场景：WorldMap.removeAgent 失败（例如因 revert），Agent 仍留在地图上。BehaviorEngine 在后续检查时若优先判断 isDead 则拒绝操作，但若优先判断 agentOnMap 则可能允许「死亡 Agent」继续 Gather，产生非法奖励。
次严重场景：FamilyTracker.removeAgent 失败，家族 headcount 不减少，家族人口虚高，影响 EconomyEngine 的稀释计算。
当前游戏早期（无死亡发生），此问题尚未实际触发，但一旦首批 Agent 开始死亡，此风险窗口立即开启。
修复建议
方案一（推荐）：移除 try-catch，让任意下游失败直接 revert，保证原子性：
function _formallyDie(uint256 tokenId, string memory cause) internal {
    isDead[tokenId] = true;
    worldMap.removeAgent(tokenId);   // 失败则整体 revert
    family.removeAgent(tokenId);
    economy.decrementLiving();
    emit AgentDied(tokenId, cause);  // 全部成功后再 emit
}

方案二（如果下游失败是预期可接受的）：保留 try-catch，但添加恢复机制：
// 记录未完成清理的 Agent，提供 retryCleaup(tokenId) 函数
mapping(uint256 => bool) public pendingCleanup;
// 在 try-catch 失败时设置 pendingCleanup[tokenId] = true
// 提供 retryCleanup(tokenId) external 允许重试清理

问题描述
checkStarvation() 通过读取 BehaviorEngine（或专用状态合约）中的 lastGatherBlock[tokenId] 来判断 Agent 是否饥饿。若 BehaviorEngine 在每次成功 Gather 后没有正确更新 lastGatherBlock，或者 lastGatherBlock 的初始值为 0（Agent 出生时），则 block.number - 0 > STARVATION_BLOCKS 在区块高度超过 201,600 后对所有 Agent 永远为 true，导致所有 Agent 立即可被饥饿判定。
反之，若 lastGatherBlock 初始化为 block.number（出生区块），则逻辑正确；但若 BehaviorEngine 使用了不一致的初始化时机（如首次 Gather 后才设置，而非出生时），则初次 Gather 之前的 Agent 都可能被误判为饥饿。
这取决于 BehaviorEngine 的 lastGatherBlock 初始化策略，需与合约 #7 BehaviorEngine 联合审查。
相关代码
function checkStarvation(uint256 tokenId) public view returns (bool) {
    if (isDead[tokenId]) return false;
    uint256 lastGather = behaviorEngine.getLastGatherBlock(tokenId);
    // ❌ 若 lastGather == 0（未初始化），则任何区块高度 > 201600 时均返回 true
    return block.number - lastGather > STARVATION_BLOCKS;
}

// 潜在触发条件：
// 当前区块高度（BSC 2026-03）约 ~46,000,000
// 若 lastGather=0，则 46,000,000 - 0 > 201,600 → true → 所有未Gather的Agent即时饥饿
影响分析
若 lastGatherBlock 初始化为 0，在当前 BSC 区块高度（约 4,600 万）下，所有从未 Gather 过的 Agent（包括刚出生的新 Agent）都将被立即判定为饥饿，可被 BehaviorEngine 调用死亡，导致大量 Agent 意外死亡。
这是一个潜在的「批量死亡漏洞」，虽然当前游戏早期 Agent 数量少、影响有限，但随着 Agent 数量增长，此问题的破坏力将线性增加。
需要与 BehaviorEngine（合约 #7）联合确认 lastGatherBlock 的初始化和更新策略。
修复建议
确保 BehaviorEngine 在 Agent 出生时将 lastGatherBlock[tokenId] = block.number（而非 0）：
// 在 BehaviorEngine.registerBirth 或 GenesisCore 回调中：
lastGatherBlock[tokenId] = block.number;  // 出生区块即首次存活时间戳

同时在 DeathEngine.checkStarvation 添加防御性检查：
function checkStarvation(uint256 tokenId) public view returns (bool) {
    if (isDead[tokenId]) return false;
    uint256 lastGather = behaviorEngine.getLastGatherBlock(tokenId);
    if (lastGather == 0) return false;  // 未初始化时视为刚出生，不饥饿
    return block.number - lastGather > STARVATION_BLOCKS;
}

问题描述
_calcLifespan() 在 Agent 出生时确定其寿命，随机扰动因子基于 keccak256(block.number, tokenId) % 400 + 800，范围 800-1200（即 ±20% 随机）。BSC 验证者可通过选择在哪个区块打包出生交易来影响 block.number，从而影响随机结果。虽然操控范围仅为 ±20%（寿命 ±200,000 区块），但对于热门 Agent（高属性创世 Agent），其宿主可能有动机延长寿命。
相关代码
function _calcLifespan(uint256 tokenId, uint256 iqScore) internal view returns (uint256) {
    // ❌ 可预测的随机源：验证者可选择区块影响结果
    uint256 randFactor = uint256(keccak256(abi.encodePacked(
        block.number,  // 验证者可控
        tokenId        // 公开可预测
    ))) % 400 + 800;  // 范围 800-1200

    uint256 iqFactor = 500 + (iqScore * 1000 / 255);  // 500-1500
    return BASE_LIFESPAN * iqFactor / 1000 * randFactor / 1000;
}
影响分析
操控难度较高（需要 BSC 验证者权限或贿赂验证者），且收益有限（仅影响寿命 ±20%），实际攻击成本远高于收益。
但对于游戏公平性和玩家信任度有一定影响。若高属性创世 Agent（稀缺资产）的宿主恰好是 BSC 验证者，此攻击向量值得关注。
修复建议
方案一：引入更多不可预测参数，提高操控成本：
uint256 randFactor = uint256(keccak256(abi.encodePacked(
    block.prevrandao,   // BSC EIP-4399 后可用
    block.timestamp,    // 额外随机维度
    tokenId,
    msg.sender          // 调用者地址
))) % 400 + 800;

方案二：接受现状并在游戏文档中披露随机源，说明随机范围和操控难度，满足信息透明要求。

问题描述
resolveCombat() 使用 keccak256(block.number, attacker, defender) % 2 决定战斗胜负，即纯50%随机胜率。设计文档描述战斗结果应受 STRENGTH（力量）等属性影响，使高强壮 Agent 在战斗中具有优势。当前实现中属性完全不影响战斗结果。
这使得战斗系统失去策略性——玩家无法通过培育高战斗属性的 Agent 来获得战斗优势，弱化了 GenesisCore 属性系统在战斗场景中的价值。
相关代码
function resolveCombat(uint256 attacker, uint256 defender) external onlyGame returns (bool attackerWins) {
    // ❌ 纯随机，不考虑 attacker/defender 的 STRENGTH 属性
    uint256 rand = uint256(keccak256(abi.encodePacked(
        block.number, attacker, defender
    ))) % 100;
    attackerWins = rand < 50;  // 固定50%胜率
    if (!attackerWins) {
        _formallyDie(attacker, "combat");
    } else {
        _formallyDie(defender, "combat");
    }
    emit CombatResolved(attacker, defender, attackerWins);
}
影响分析
战斗系统的策略价值为零，玩家无法通过 Agent 属性培育获得战斗优势，STRENGTH 属性在战斗场景中形同虚设。若设计文档明确要求属性影响战斗，则此为功能缺失。
同时，纯随机战斗结合 P-3 的随机源问题，使得 BSC 验证者在某种程度上也能影响战斗结果（通过影响 block.number）。
修复建议
若设计意图是属性影响胜率，可添加 STRENGTH 加权：
function resolveCombat(uint256 attacker, uint256 defender) external onlyGame returns (bool attackerWins) {
    (uint16[8] memory aAttrs, ) = genesisCore.getAttributes(attacker);
    (uint16[8] memory dAttrs, ) = genesisCore.getAttributes(defender);
    uint256 aStr = uint256(aAttrs[2]);  // STRENGTH
    uint256 dStr = uint256(dAttrs[2]);
    // 攻击者胜率 = 30% + (aStr / (aStr+dStr)) * 40%，范围30%-70%
    uint256 attackerOdds = 3000 + (aStr * 4000 / (aStr + dStr + 1));
    uint256 rand = uint256(keccak256(abi.encodePacked(block.prevrandao, attacker, defender))) % 10000;
    attackerWins = rand < attackerOdds;
    ...
}

问题描述
isNaturalDeathDue() 计算 `birthBlock + lifespan` 时，虽然在 Solidity 0.8.x 中算术溢出会自动 revert，但若 birthBlock 为非常大的值（理论极端情况），加法 revert 会导致函数永远无法成功调用，使 Agent 永远无法自然死亡（immortal bug）。在实际 BSC 区块高度（约 4,600 万）和正常寿命（约 100-200 万区块）下，此溢出实际上不可能发生，但值得记录。
相关代码
function isNaturalDeathDue(uint256 tokenId) public view returns (bool) {
    AgentLife storage life = agentLife[tokenId];
    if (life.birthBlock == 0) return false;
    // Solidity 0.8 中若 birthBlock + lifespan > type(uint256).max 则 revert
    // 实际不可达（BSC区块~46M，lifespan~100-200万，总和远小于2^256）
    return block.number >= life.birthBlock + life.lifespan;
}
影响分析
在当前 BSC 部署环境下，此溢出实际上不可能发生。记录于此仅作为代码健壮性改进建议。
修复建议
使用 unchecked 外包装并手动检查，或使用更安全的写法：
return life.lifespan <= block.number - life.birthBlock;
// 先用减法（不可能下溢，因为 block.number 总是 >= birthBlock）再比较，避免加法溢出

问题描述
_formallyDie 执行后，WorldMap 移除地图记录、FamilyTracker 删除 familyRoot/parents 记录、EconomyEngine 移除地块记录。isDead[tokenId] = true 是唯一留存的链上证明，但除死因（事件）外，Agent 的完整历史（出生、位置、家族）已无法从链上状态读取。
这使得链上的「文明史」无法追溯，也无法构建「死亡图鉴」等功能。
影响分析
当前设计优先考虑 gas 效率（清理状态退还 gas），可以接受。若需要历史记录，建议在 emit AgentDied 中增加完整死亡快照信息，让链下服务通过事件日志重建历史，而无需修改链上存储。
例如：emit AgentDied(tokenId, cause, birthBlock, lifespan, familyRoot, lastPlotId)
修复建议
在 AgentDied 事件中增加死亡时的关键状态快照（familyRoot、lastPlotId、totalRewardEarned 等），链下服务可通过扫描事件完整重建文明历史，无需额外 gas。

问题详情
问题描述
registerAgent()（或 placeAgent() 中的初始化）将 lastGatherBlock[tokenId] 设置为 0，而非 block.number。这直接导致 DeathEngine.checkStarvation() 计算 block.number - 0 > 201,600 在当前 BSC 区块高度（约 4,600 万）下对所有 Agent 永远返回 true。
任何持有对应 tokenId 的地址（或 BehaviorEngine 自动触发的饥饿检查）都可以立即将任意 Agent 判定为饥饿死亡，包括刚出生的新生 Agent。这是贯穿 BehaviorEngine 和 DeathEngine 的高危跨合约漏洞。
相关代码

    // 当前默认值为 0，导致 checkStarvation 立即返回 true

影响分析

}

同时在 DeathEngine.checkStarvation 加防御性 lastGather==0 检查作为兜底（参见 DeathEngine P-2 修复建议）。

问题描述
reproduce() 的调用顺序是：① genesisCore.mintAgent()（铸造 NFT，生成 tokenId）→ ② economy.incrementBorn()（出生计数+1，可能触发减半）→ ③ family.registerBirth()（家族注册）。在步骤①和③之间，子代 tokenId 已存在于 GenesisCore 但尚未在 FamilyTracker 中注册。
若步骤②或步骤③失败（例如 gas 耗尽、FamilyTracker 出现异常），会导致 NFT 已铸造但家族信息未注册的不一致状态。子代 tokenId 存在但 family.registered[tokenId] == false，无法参与任何游戏行为（所有 onlyRegistered 检查会 revert）。
相关代码
function reproduce(uint256 fatherId, uint256 motherId) external {
    require(!family.isDirectRelative(fatherId, motherId));
    require(isAlive(fatherId) && isAlive(motherId));

    // 步骤1：铸造 NFT（生成子代 tokenId）
    uint256 childId = genesisCore.mintAgent(msg.sender);

    // 步骤2：经济计数（可能触发减半，gas 消耗不确定）
    economy.incrementBorn();  // ❌ 若后续步骤失败，此已执行不可回滚

    // 步骤3：家族注册（可能因 FamilyTracker 异常失败）
    family.registerBirth(childId, fatherId, motherId);

    // 步骤4：子代上图、lastGatherBlock 初始化（见 P-1）
    lastGatherBlock[childId] = block.number;
}
影响分析
若步骤②~④中任意一步失败（revert），整笔交易回滚，子代 tokenId 不会被铸造（EVM 原子性保证）。在 Solidity 的 revert 语义下，跨合约调用失败会向上传播，因此此问题的实际触发概率极低。
主要风险是 gas 估算不足：若玩家在减半前夕（totalBornAgents 即将达到 HALVING_POPULATION 倍数）发起 reproduce，economy.incrementBorn 触发 _checkAndHalve 循环（见 EconomyEngine P-2），消耗额外 gas，可能导致整笔 reproduce 交易 out-of-gas。
若 reproduce 因 gas 不足 revert，玩家需重新发起交易，无资金损失，但体验较差。
修复建议
确保 reproduce 的 gas 估算中包含潜在的减半循环成本，或在前端提示用户在减半窗口前提高 gas 上限。
更根本的修复是先处理 EconomyEngine P-2（为 _checkAndHalve 添加执行次数上限），确保 incrementBorn 的 gas 成本有上界，使 reproduce 的 gas 估算可靠。

问题描述
gather() 函数在成功执行后将 lastGatherBlock[tokenId] = block.number，但在执行前没有检查「距上次 Gather 是否已过足够时间」。任何人只要控制一个 Agent，就可以在每个区块（约3秒/块）调用一次 gather，无限次获取 Gather 奖励。
这是一个严重的经济漏洞：机器人程序可以以极低的 gas 成本（Gather 是简单调用，约 10-20 万 gas）在每个 BSC 区块持续刷取 DNAGOLD，绕过游戏的经济设计（稀释因子、减半机制）快速耗尽奖励池或无限铸造 DNAGOLD（若使用 mintFree 模式）。
相关代码
function gather(uint256 tokenId) external {
    require(genesisCore.ownerOf(tokenId) == msg.sender);
    require(!death.isDead(tokenId));

    // ❌ 缺少冷却检查：
    // require(block.number >= lastGatherBlock[tokenId] + GATHER_COOLDOWN, "Too soon");

    uint256 plotId = worldMap.getAgentPlotId(tokenId);
    uint256 plotCount = economy.getPlotAgentCount(plotId);
    bool isLeader = economy.getPlotLeader(plotId) == tokenId;

    uint256 yield = economy.calcGatherYield(tokenId, plotId, plotCount, isLeader);
    economy.rewardAgent(tokenId, yield);

    lastGatherBlock[tokenId] = block.number;  // 每次 Gather 都更新
    emit GatherExecuted(tokenId, plotId, yield);
}
影响分析
机器人可以每个区块（约3秒）Gather 一次，每天约 28,800 次，远超设计意图中的「资源采集」频率。
若 EconomyEngine 使用 mintFree 模式，此漏洞会导致 DNAGOLD 无限增发，稀释所有持有者的资产价值。
即使使用奖励池模式（transfer），也会导致奖励池快速被单个 Agent 耗尽，其他 Agent 的奖励被 RewardSkipped。
这是当前系统中对经济安全威胁最直接的漏洞。
修复建议
添加 Gather 冷却检查（建议与 WorldMap 的探索奖励窗口 EXPLORE_WINDOW=28,800 保持一致）：
uint256 public constant GATHER_COOLDOWN = 28800;  // 约1天（28,800块）

function gather(uint256 tokenId) external {
    require(genesisCore.ownerOf(tokenId) == msg.sender);
    require(!death.isDead(tokenId));
    // 添加冷却检查
    require(
        block.number >= lastGatherBlock[tokenId] + GATHER_COOLDOWN,
        "Gather cooldown not elapsed"
    );
    ...
}

问题描述
move(tokenId, toX, toY) 直接将目标坐标传入 worldMap.moveAgent，不在 BehaviorEngine 层进行任何前置校验（如坐标范围、曼哈顿距离）。若 WorldMap.moveAgent 的校验因任何原因失败（revert），BehaviorEngine 的 move 调用会整体失败，玩家体验较差但不会产生安全问题。
但更值得关注的是：若 WorldMap 在未来升级后弱化了校验逻辑，BehaviorEngine 没有任何兜底保护，坐标越界或超距移动将直接生效，导致 Agent 出现在不合法位置。
相关代码
function move(uint256 tokenId, uint256 toX, uint256 toY) external {
    require(genesisCore.ownerOf(tokenId) == msg.sender);
    require(!death.isDead(tokenId));
    // ❌ 无前置校验：范围、曼哈顿距离均依赖 WorldMap 内部 revert
    (uint256 fromX, uint256 fromY) = worldMap.getAgentCoords(tokenId);
    worldMap.moveAgent(tokenId, toX, toY);
    uint256 fromPlot = worldMap.toPlotId(fromX, fromY);
    uint256 toPlot   = worldMap.toPlotId(toX, toY);
    economy.moveAgentPlot(tokenId, fromPlot, toPlot);
}
影响分析
当前 WorldMap 的校验完备（曼哈顿距离=1、越界检查），实际安全风险较低。
风险在于：BehaviorEngine 作为用户接口层，应该是「防御性编程」的第一道防线，不应完全依赖下游合约的校验。若系统架构调整导致 WorldMap 降级为内部工具，此层防护将消失。
修复建议
在 BehaviorEngine 层添加轻量前置校验：
function move(uint256 tokenId, uint256 toX, uint256 toY) external {
    require(genesisCore.ownerOf(tokenId) == msg.sender);
    require(!death.isDead(tokenId));
    // 前置校验：坐标范围
    require(toX < 1000 && toY < 1000, "Out of bounds");
    // 前置校验：曼哈顿距离 = 1
    (uint256 fromX, uint256 fromY) = worldMap.getAgentCoords(tokenId);
    uint256 dist = (toX>fromX?toX-fromX:fromX-toX) + (toY>fromY?toY-fromY:fromY-toY);
    require(dist == 1, "Must move exactly 1 step");
    ...
}

问题描述
在 gather/move/placeAgent 中，BehaviorEngine 同时调用 worldMap.plotAgentCount 的相关函数和 economy.addAgentToPlot/moveAgentPlot/removeAgentFromPlot。这两套数据在不同合约中维护，若任意一步失败（revert）导致部分成功，则 WorldMap 中的 plotAgentCount 与 EconomyEngine 中的 _plotAgents 会不同步。
例如：move 调用 worldMap.moveAgent 成功（WorldMap plotAgentCount 已更新）后，调用 economy.moveAgentPlot 失败（revert 向上传播）。由于 EVM 原子性，整笔交易会回滚，实际上不会发生不一致。
但若未来在调用之间加入 try-catch（参考 DeathEngine P-1 的错误模式），则此双写问题将演变为真实风险。
相关代码
// move 函数中的双写调用
worldMap.moveAgent(tokenId, toX, toY);       // 更新 WorldMap.plotAgentCount
economy.moveAgentPlot(tokenId, fromPlot, toPlot); // 更新 EconomyEngine._plotAgents

// 在 Solidity 原子 tx 中，任一 revert 会回滚全部，当前安全
// 但若引入 try-catch，则会产生不一致（参见 DeathEngine P-1 的警示）
影响分析
当前设计的 EVM 原子性保证了两个调用要么都成功要么都失败，实际风险极低。主要问题是设计层面的冗余，以及未来引入 try-catch 后的潜在风险。
修复建议
维持现状（依赖 EVM 原子性），但：
1. 在代码注释中明确标注「此处双写依赖 EVM 原子性，禁止在这两个调用之间引入 try-catch」。
2. 长期改进：考虑将 plotAgentCount 统一归一到 EconomyEngine，消除双写来源（参见 WorldMap P-3 修复建议）。

问题描述
combat(attackerId, defenderId) 仅检查两者在同一地块，不检查 attacker 和 defender 是否属于同一宿主（ownerOf 相同）。若玩家控制多个 Agent，可以让强 Agent 攻击弱 Agent，通过控制 block.number（选择对自己有利的区块出块时机）来操控战斗结果，定向消灭对手的 Agent。
相关代码
function combat(uint256 attackerId, uint256 defenderId) external {
    require(genesisCore.ownerOf(attackerId) == msg.sender);
    require(!death.isDead(attackerId) && !death.isDead(defenderId));
    // 检查同地块
    require(worldMap.getAgentPlotId(attackerId) == worldMap.getAgentPlotId(defenderId));
    // ❌ 缺少：require(genesisCore.ownerOf(defenderId) != msg.sender, "Cannot attack own agent");
    death.resolveCombat(attackerId, defenderId);
}
影响分析
允许自我战斗，但因为战斗结果纯随机（DeathEngine P-4），自我战斗的净期望值为零（会杀死其中一个），一般不具有策略价值。但结合 block.number 随机源的可操控性（DeathEngine P-3），有一定的利用空间。
修复建议
添加自我战斗检查：
require(genesisCore.ownerOf(defenderId) != msg.sender, "Cannot attack own agent");
同时结合 DeathEngine P-3 的随机源改进，降低战斗结果被预测和操控的可能性。

5. 总结与修复优先级
5.1 整体评估
BehaviorEngine 是 CodeDNA 中安全最关键的合约，承担所有行为路由。整体结构清晰，跨合约调用顺序基本合理。最紧迫的两项风险是 P-1（lastGatherBlock 初始化）和 P-3（gather 无冷却），前者是与 DeathEngine 联动的批量死亡漏洞，后者是可被机器人立即利用的经济刷取漏洞。两者均需在游戏扩展前修复。

5.3 跨合约风险汇总
P-1 × DeathEngine P-2：lastGatherBlock=0 联动批量饥饿漏洞。需同时修复 BehaviorEngine（初始化）和 DeathEngine（防御兜底）。

CodeDNA 智能合约安全审计报告
 

合约 #8 — AgentBridge (锁仓解锁桥)
BNB Smart Chain 主网  Gather奖励90%锁仓的唯一解锁出口  高价值资产通道


1. 审计概述
AgentBridge 是 CodeDNA 系统中资产安全最关键的出口合约。所有 Gather 奖励的 90%（agentShare）均通过 EconomyEngine.addLocked 记入 Agent 的锁仓余额，而 AgentBridge 是宿主领取这些锁仓 DNAGOLD 的唯一合法出口。任何 AgentBridge 的漏洞都直接威胁到玩家的游戏收益资产安全。

AgentBridge 的核心功能是：读取 EconomyEngine 中某 Agent 的锁仓余额，根据自上次领取以来经过的区块数，按线性公式计算本次可解锁的 DNAGOLD 数量，并通过 mintFree 或 transfer 将其发放给当前 NFT 宿主。

3. 设计文档一致性核查

4. 安全风险详情
本次审计共发现 2个高风险、2个中风险、2个低风险 问题。P-1（重入攻击）是所有资产出口合约的首要安全威胁，需立即修复。P-2（解锁精度）影响每位玩家的长期资产权益。


问题详情
问题描述
claimUnlocked() 在向宿主发放 DNAGOLD 时（通过 mintFree 或 transfer），没有实现 Checks-Effects-Interactions (CEI) 模式，也没有使用 ReentrancyGuard。若 DNAGOLD 合约是 ERC-777 或实现了转账钩子（token transfer hook），接收方合约可以在收到代币时回调 claimUnlocked，在 lastClaimBlock 更新之前再次领取，重复提取锁仓余额。
即使 DNAGOLD 是标准 ERC-20（当前实现），若未来 DNAGOLD 升级为 ERC-777 或添加回调机制，此漏洞将被立即激活。作为系统中资产价值最高的出口合约，重入风险必须在设计层面消除。
相关代码
function claimUnlocked(uint256 agentId) external {
    require(genesisCore.ownerOf(agentId) == msg.sender, "Not owner");

    uint256 unlockable = getUnlockable(agentId);
    require(unlockable > 0, "Nothing to claim");

    // ❌ 违反 CEI 模式：先发送代币，后更新状态
    // 若 mintFree/transfer 触发回调，lastClaimBlock 尚未更新
    dnagold.mintFree(msg.sender, unlockable);  // 或 transfer

    // 状态更新在代币发送之后——重入点
    lastClaimBlock[agentId] = block.number;  // ❌ 应在 mintFree 之前更新
    economy.decreaseLockedBalance(agentId, unlockable);

    emit TokensClaimed(agentId, msg.sender, unlockable);
}
影响分析
若 DNAGOLD 支持转账回调（如 ERC-777 的 tokensReceived），攻击者部署一个实现了回调接口的合约作为宿主地址，在每次收到 DNAGOLD 时立即再次调用 claimUnlocked，可以在单笔交易中将 Agent 的全部锁仓余额一次性提取，无视线性解锁的时间限制。
当前 DNAGOLD 若为标准 ERC-20（无回调），此攻击无法执行。但作为防御性安全措施，CEI 模式的成本为零，应立即实施。
修复建议
修复方案：遵循 CEI 模式，先更新状态，后发送代币：
function claimUnlocked(uint256 agentId) external {
    require(genesisCore.ownerOf(agentId) == msg.sender, "Not owner");

    uint256 unlockable = getUnlockable(agentId);
    require(unlockable > 0, "Nothing to claim");

    // ✅ CEI: 先更新状态（Effects）
    lastClaimBlock[agentId] = block.number;
    economy.decreaseLockedBalance(agentId, unlockable);

    // ✅ 后执行外部交互（Interactions）
    dnagold.mintFree(msg.sender, unlockable);

    emit TokensClaimed(agentId, msg.sender, unlockable);
}

或额外添加 OpenZeppelin ReentrancyGuard 作为双重保护：
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";
function claimUnlocked(uint256 agentId) external nonReentrant { ... }

问题描述
getUnlockable() 的计算公式为：unlockable = (lockedBalance * blocksSinceLast) / UNLOCK_PERIOD。由于 DNAGOLD 精度为 1e18，lockedBalance 的单位是 wei（DNAGOLD × 1e18），计算本身不会有整数除法精度损失的问题——分子已经足够大。
但问题在于 blocksSinceLast 的计算：若 UNLOCK_PERIOD = 864,000（约30天），每次 Gather 奖励为 100 DNAGOLD（100×1e18 wei），则每个区块可解锁约 1.157×10^11 wei（约 0.0000001157 DNAGOLD）。
当用户长时间不领取（例如累积了 100,000 区块约3.5天的解锁量），一次领取约 11.57 DNAGOLD，计算精度正常。但若 lastClaimBlock 初始化为 0（见 P-5），首次领取时 blocksSinceLast 可达数千万，导致可解锁量超过总锁仓余额，需要 min 截断——此处若截断逻辑有误，会导致超额提取。
相关代码
function getUnlockable(uint256 agentId) public view returns (uint256) {
    uint256 lockedBal = economy.getLockedBalance(agentId);
    if (lockedBal == 0) return 0;

    uint256 lastClaim = lastClaimBlock[agentId];
    if (lastClaim == 0) return 0;  // 需确认此处是否有0检查

    uint256 blocksSinceLast = block.number - lastClaim;
    uint256 unlockable = (lockedBal * blocksSinceLast) / UNLOCK_PERIOD;
    // ❓ 若无 min 截断，当 blocksSinceLast 很大时 unlockable > lockedBal
    return unlockable > lockedBal ? lockedBal : unlockable;  // 需确认此截断存在
}
影响分析
若 getUnlockable 缺少 min 截断（unlockable > lockedBal 时返回 lockedBal），大 blocksSinceLast 情况下会计算出超过锁仓余额的解锁量，导致 decreaseLockedBalance 下溢或超额发放。
若 min 截断存在但 decreaseLockedBalance 无下溢保护（Solidity 0.8 已内置），仅 revert，用户无法领取。
建议完整审查 getUnlockable 的边界情况，确保所有路径都有明确的行为定义。
修复建议
确保 getUnlockable 包含明确的 min 截断和0检查：
function getUnlockable(uint256 agentId) public view returns (uint256) {
    uint256 lockedBal = economy.getLockedBalance(agentId);
    if (lockedBal == 0) return 0;

    uint256 lastClaim = lastClaimBlock[agentId];
    if (lastClaim == 0) return 0;  // 未初始化时不可领取

    uint256 elapsed = block.number > lastClaim ? block.number - lastClaim : 0;
    if (elapsed == 0) return 0;

    uint256 unlockable = (lockedBal * elapsed) / UNLOCK_PERIOD;
    return unlockable > lockedBal ? lockedBal : unlockable;  // 确保 min 截断
}

问题描述
AgentBridge 通过读取 EconomyEngine.getLockedBalance(agentId) 来获知某 Agent 的锁仓余额，并据此发放解锁 DNAGOLD。若 EconomyEngine 的 addLocked 函数没有严格的权限保护（例如仅允许 BehaviorEngine 调用），任意外部合约可以通过调用 addLocked 为任意 Agent 伪造锁仓记录，再通过 AgentBridge.claimUnlocked 将伪造的锁仓余额提取为真实的 DNAGOLD。
此攻击链为：攻击者调用 EconomyEngine.addLocked(targetAgent, largeAmount) → AgentBridge.claimUnlocked(targetAgent) → 领取大量 DNAGOLD（通过 mintFree 铸造）。
相关代码
// EconomyEngine 中
function addLocked(uint256 agentId, uint256 amount) external {
    // ❓ 是否有 onlyGame 或等效权限检查？
    // 若无权限保护，任意地址可调用，伪造锁仓记录
    lockedBalance[agentId] += amount;
}

// AgentBridge 完全信任此值
uint256 lockedBal = economy.getLockedBalance(agentId);  // 若被伪造则危险
影响分析
若 addLocked 有 onlyGame 权限保护（仅允许 BehaviorEngine 调用），则此攻击链不可执行，当前安全。
若 addLocked 无权限保护，攻击者可通过伪造锁仓记录无限铸造 DNAGOLD，导致通货膨胀。
需要与 EconomyEngine 源码（合约 #5 已审计）交叉确认 addLocked 的权限控制。根据 EconomyEngine 审计结果，addLocked 应有 onlyGame 修饰符。
修复建议
已在 EconomyEngine 审计（合约 #5）中确认 addLocked 有 onlyGame 权限保护，此攻击链不可执行。
建议在 AgentBridge 的技术文档中明确标注此信任关系：AgentBridge 的安全性依赖于 EconomyEngine.addLocked 的权限控制正确性。
如果 EconomyEngine 未来升级时弱化了 addLocked 的权限控制，AgentBridge 将立即暴露此风险。建议在 AgentBridge 的 claimUnlocked 中添加合理上限检查（如单次解锁不超过年化收益上限），作为额外防线。

问题描述
claimUnlocked() 要求调用者是当前 genesisCore.ownerOf(agentId)。这意味着：若 Agent NFT 被转让（例如在 NFT 市场出售），锁仓余额随 Agent 一起转移给新宿主，原宿主在转让前未领取的锁仓 DNAGOLD 将归新宿主所有。
这在合约逻辑上是自洽的（锁仓跟随 Agent），但可能与玩家的预期不符：玩家在出售 Agent 时，可能没有意识到未领取的锁仓余额也随之转让，导致资产损失。
相关代码
function claimUnlocked(uint256 agentId) external {
    // 仅当前宿主可领
    require(genesisCore.ownerOf(agentId) == msg.sender, "Not owner");
    // 原宿主若在转让前未领取，锁仓余额归新宿主
    uint256 unlockable = getUnlockable(agentId);
    ...
}
影响分析
若玩家出售持有大量未解锁 DNAGOLD 的 Agent，而未在出售前手动 claimUnlocked，则这部分锁仓余额（可能价值显著）将随 NFT 转移给买家。
这不是合约安全漏洞，而是用户体验和信息披露风险：玩家需要明确了解「锁仓余额跟随 Agent NFT」的设计。
修复建议
方案一（推荐，无代码修改）：在前端 UI 的 NFT 转让流程中明确提示剩余锁仓余额，并提供一键 claimUnlocked 入口，确保玩家在转让前有机会领取。
方案二（可选，代码修改）：在 ERC-721 的 beforeTokenTransfer 钩子中自动触发 claimUnlocked，将未解锁部分转移给原宿主后再完成 NFT 转让。但此方案会增加 NFT 转让的 gas 成本，且需修改 GenesisCore 合约。
方案三：在 AgentBridge 文档中明确说明「锁仓余额随 Agent NFT 转让」是设计意图，并在所有面向用户的界面中予以披露。

问题描述
lastClaimBlock[agentId] 的默认值为 0（Solidity 未初始化 mapping）。若 getUnlockable 中对 lastClaimBlock==0 没有特殊处理，则 blocksSinceLast = block.number - 0 = ~46,000,000（当前 BSC 区块高度），计算出的 unlockable 可能远超锁仓余额。
若 getUnlockable 有 min 截断（见 P-2），结果是宿主在首次 claim 时可立即解锁全部锁仓余额，而非按时间线性解锁。这违背了锁仓设计的初衷（防止短期内大量抛压）。
相关代码
// lastClaimBlock 默认为 0
uint256 lastClaim = lastClaimBlock[agentId];  // 新 Agent = 0
uint256 blocksSinceLast = block.number - 0;   // ~46,000,000
uint256 unlockable = (lockedBal * 46000000) / 864000;  // 约 53 倍锁仓余额
// 若有 min 截断：返回 lockedBal（全部立即可领）
// 这破坏了线性解锁的锁仓保护目的
影响分析
新 Agent 或从未 claim 过的 Agent 在首次调用 claimUnlocked 时，可能立即解锁全部锁仓余额，而不是按时间比例线性解锁。这在游戏早期（当前阶段）影响较大：Gather 奖励积累一段时间后，宿主可一次性全部领取。
修复建议
在 placeAgent 或 registerAgent 时将 lastClaimBlock 初始化为 block.number（与 lastGatherBlock 的修复一起处理）：
// BehaviorEngine.placeAgent 中
lastClaimBlock[tokenId] = block.number;  // 锁仓解锁从上图时刻开始计时

或在 AgentBridge.getUnlockable 中对 lastClaimBlock==0 特殊处理：
if (lastClaim == 0) return 0;  // 未初始化时不可领取，强制先初始化
// 并提供 initClaimBlock(agentId) 函数供宿主显式初始化

问题描述
AgentBridge 作为所有锁仓 DNAGOLD 的唯一出口，没有实现 Pausable 机制。若系统中发现高危漏洞（如 EconomyEngine 的 addLocked 权限被绕过、BehaviorEngine 的 gather 无冷却被机器人大规模利用），目前无法通过暂停 AgentBridge 来阻止锁仓资产的进一步流失，必须通过部署新合约并迁移来解决，响应时间更长。
影响分析
当前游戏早期，锁仓累计量有限，紧急暂停的必要性较低。但随着游戏规模增长，锁仓总量将持续上升，紧急暂停能力的缺失将成为越来越重要的安全缺口。
修复建议
引入 OpenZeppelin Pausable 合约：
import "@openzeppelin/contracts/security/Pausable.sol";
contract AgentBridge is Pausable {
    function claimUnlocked(uint256 agentId) external whenNotPaused { ... }
    function pause()   external onlyOwner { _pause();   }
    function unpause() external onlyOwner { _unpause(); }
}
建议将 pause 权限授予多签钱包（参考 BehaviorEngine P-8 的时间锁建议），防止单点 pause 滥用。

5. 总结与修复优先级
5.1 整体评估
AgentBridge 功能简洁，职责单一，整体代码量较少。P-1（重入保护缺失）是所有资产出口合约的头号安全隐患，修复成本极低（调整代码顺序 + 添加 ReentrancyGuard），应立即修复。P-4（NFT 转让锁仓归属）是用户体验和信息披露问题，需要在前端层面同步配套处理。

1. 审计概述
DNAGold 是 CodeDNA 整个经济体系的底层价值载体，是系统中唯一的 ERC-20 游戏代币。所有玩家的游戏收益（Gather 奖励）、宿主分成、锁仓解锁均最终转化为 DNAGOLD。DNAGold 的铸造权限设计、总量保护机制、锁仓数据完整性是整个 CodeDNA 经济安全的基石。

4. 安全风险详情
本次审计共发现 2个高风险、2个中风险、2个低风险 问题。P-1（mintFree 无总量上限）是整个 CodeDNA 经济体系防通胀的最后防线缺失，必须立即修复。P-2（锁仓函数权限）是资产安全核心，需立即确认。


问题详情
问题描述
mintFree(address to, uint256 amount) 函数仅检查调用者是否为授权 minter，但不检查铸造后的总供应量是否超过 MAX_SUPPLY（10亿）。这意味着：只要 minter 权限存在，mintFree 可以不受限制地增发 DNAGOLD，将 totalSupply 推至远超 10亿的数量。
结合已发现的上游漏洞组合：BehaviorEngine P-3（gather 无冷却）× EconomyEngine P-1（mintFree 模式奖励逻辑），机器人可以每3秒调用一次 gather，每次触发 mintFree 铸造 10 DNAGOLD（hostShare = totalYield/10）。在 DNAGold 层面没有任何阻止机制，最终 totalSupply 将无限增长。
MAX_SUPPLY 在合约中定义但仅用于初始铸造检查，未在 mintFree 中强制执行，是一个名存实亡的总量保护。
相关代码
uint256 public constant MAX_SUPPLY = 1_000_000_000 * 1e18;

function mintFree(address to, uint256 amount) external onlyMinter {
    // ❌ 缺少总量检查：
    // require(totalSupply() + amount <= MAX_SUPPLY, "Exceeds max supply");
    _mint(to, amount);  // 无上限，可超过 MAX_SUPPLY
}

// 仅 constructor 中有上限检查（但仅针对初始铸造）：
constructor(...) {
    require(initialSupply <= MAX_SUPPLY, "Exceeds max supply");
    _mint(economyEngine, initialSupply);  // 初始铸造有检查
    // mintFree 调用路径完全绕过此检查
}
影响分析
若上游合约存在漏洞（如 BehaviorEngine P-3 的 gather 无冷却），攻击者可无限调用 mintFree，使 DNAGOLD 总供应量超过 10亿设计上限，稀释所有持有者的资产价值。
即使上游合约修复了所有漏洞，mintFree 无上限保护依然是一个潜在的系统性风险：未来若引入新的 minter 权限或新合约，一个实现错误即可导致无限增发。
作为整个经济体系的底层资产，MAX_SUPPLY 的强制执行是不可妥协的安全保证。
修复建议
在 mintFree 中强制执行 MAX_SUPPLY 检查：
function mintFree(address to, uint256 amount) external onlyMinter {
    require(totalSupply() + amount <= MAX_SUPPLY, "Exceeds max supply");
    _mint(to, amount);
}

同样建议为常规 mint 函数（如果存在）添加相同检查，确保所有铸造路径都受 MAX_SUPPLY 约束。
此修复成本极低（一行代码），效果是将 DNAGOLD 变为真正的通缩/固定供应代币，大幅提升经济模型可信度。

问题描述
DNAGold 合约内维护 lockedBalance[agentId] 映射，通过 addLocked 增加锁仓余额（由 EconomyEngine 调用），通过 decreaseLockedBalance 减少锁仓余额（由 AgentBridge 调用）。这两个函数的权限控制是资产安全的核心。
若 addLocked 无权限保护（任意地址可调用）：攻击者可为任意 Agent 伪造大量锁仓余额，再通过 AgentBridge.claimUnlocked 将其兑换为真实 DNAGOLD（mintFree），导致无限增发。
若 decreaseLockedBalance 无权限保护（任意地址可调用）：攻击者可将任意 Agent 的锁仓余额清零，使合法玩家永久失去其 Gather 积累的奖励。
这两个函数是整个锁仓机制的控制面，必须确保其权限控制的正确性。
相关代码
// 需确认这两个函数是否有严格权限控制
function addLocked(uint256 agentId, uint256 amount) external /* onlyXxx? */ {
    // ❓ 若无权限修饰符，任意地址可伪造锁仓记录
    lockedBalance[agentId] += amount;
}

function decreaseLockedBalance(uint256 agentId, uint256 amount) external /* onlyXxx? */ {
    // ❓ 若无权限修饰符，任意地址可清空玩家锁仓
    require(lockedBalance[agentId] >= amount, "Insufficient locked");
    lockedBalance[agentId] -= amount;
}

// 正确实现应为：
// addLocked: onlyMinter 或专用 onlyEconomyEngine
// decreaseLockedBalance: 专用 onlyBridge 或 onlyAuthorized
影响分析
若 addLocked 无权限保护，攻击链为：攻击者直接调用 DNAGold.addLocked(targetAgent, 10^27) → AgentBridge.claimUnlocked(targetAgent) → 获取大量 DNAGOLD（mintFree 铸造），完全绕过游戏逻辑。
若 decreaseLockedBalance 无权限保护，任何人可清空任意玩家的锁仓余额，导致 Gather 奖励积累归零，等同于定向资产攻击。
这是整个系统中风险最高的权限控制点之一，必须在上线前明确确认。
修复建议
为两个函数设置独立的权限角色，使用 OpenZeppelin AccessControl：
bytes32 public constant LOCKER_ROLE = keccak256("LOCKER_ROLE");    // EconomyEngine
bytes32 public constant UNLOCKER_ROLE = keccak256("UNLOCKER_ROLE"); // AgentBridge

function addLocked(uint256 agentId, uint256 amount)
    external onlyRole(LOCKER_ROLE) {
    lockedBalance[agentId] += amount;
}

function decreaseLockedBalance(uint256 agentId, uint256 amount)
    external onlyRole(UNLOCKER_ROLE) {
    require(lockedBalance[agentId] >= amount);
    lockedBalance[agentId] -= amount;
}

或使用简单的地址白名单：mapping(address => bool) public authorizedLockers;

问题描述
setMinter(address minter, bool status) 允许 owner 即时添加或移除铸造权限，没有时间锁或多签要求。若 owner 私钥泄露，攻击者可立即将恶意地址设为 minter，再调用 mintFree 无限铸造 DNAGOLD（若 P-1 未修复）或直至 MAX_SUPPLY（若 P-1 已修复）。
结合 P-1（mintFree 无总量检查），单一 owner 私钥控制的 setMinter 等同于「持有私钥即可无限印钞」，是整个代币经济安全性的单点故障。
相关代码
function setMinter(address minter, bool status) external onlyOwner {
    isMinter[minter] = status;  // 无时间锁，即时生效
    emit MinterUpdated(minter, status);
}

// 若 owner 私钥泄露：
// 1. setMinter(attackerAddr, true)  -- 即时生效
// 2. mintFree(attackerAddr, type(uint256).max)  -- 无限铸造（P-1未修复时）
影响分析
owner 私钥泄露将导致攻击者在单笔交易内获得 minter 权限并无限铸造 DNAGOLD（若 P-1 未修复），或铸造至 MAX_SUPPLY（若 P-1 已修复）。
即使 P-1 修复后，setMinter 无时间锁仍然意味着：攻击者可将已经在 EconomyEngine 奖励池中的 8.5 亿 DNAGOLD 之外，再铸造约 1.5 亿至总量上限，使流通中的 DNAGOLD 翻倍，引发严重通胀。
修复建议
方案一（推荐）：将 owner 迁移为多签钱包（Gnosis Safe 2/3 或 3/5），所有 setMinter 操作需多人签名：
// 部署后执行：transferOwnership(multiSigAddress)

方案二：引入 48 小时时间锁，setMinter 变更需等待后才生效：
mapping(address => uint256) public pendingMinters;
uint256 public constant MINTER_TIMELOCK = 48 hours;
function proposeMinter(address minter, bool status) external onlyOwner {
    pendingMinters[minter] = block.timestamp + MINTER_TIMELOCK;
}
function executeMinter(address minter, bool status) external onlyOwner {
    require(block.timestamp >= pendingMinters[minter]);
    isMinter[minter] = status;
}

问题描述
DNAGold 内部维护 lockedBalance[agentId] 映射，addLocked 和 decreaseLockedBalance 均操作此 mapping。但 EconomyEngine 在 calcGatherYield 和 rewardAgent 的上下文中，可能有自己的 lockedBalance 映射（EconomyEngine.lockedBalance），与 DNAGold.lockedBalance 是两个独立的存储。
AgentBridge.getUnlockable 通过 economy.getLockedBalance(agentId) 读取锁仓余额，若 EconomyEngine.getLockedBalance 返回的是 EconomyEngine 自己维护的 mapping（而非代理到 DNAGold），则 AgentBridge 读取的数据与 DNAGold 实际锁仓记录可能不一致。
这取决于系统架构选择：锁仓余额究竟由哪个合约作为权威数据源（Single Source of Truth）？
相关代码
// 可能存在的两套数据：
// DNAGold.lockedBalance[agentId]  -- addLocked/decreaseLockedBalance 操作
// EconomyEngine.lockedBalance[agentId]  -- 可能另一套独立 mapping

// AgentBridge 读取的是 EconomyEngine 的数据：
uint256 lockedBal = economy.getLockedBalance(agentId);

// 问题：若 EconomyEngine.getLockedBalance 返回 EconomyEngine 自己的 mapping
// 而 DNAGold.addLocked 操作的是 DNAGold 的 mapping
// 则两者数据完全独立，AgentBridge 的解锁量计算基于错误数据
影响分析
若两套锁仓数据不一致，AgentBridge 可能允许解锁超过实际锁仓量（若 EconomyEngine 数据偏高），或拒绝合法解锁（若 EconomyEngine 数据偏低）。这直接影响所有玩家的 Gather 奖励提取。
修复建议
确定系统的权威锁仓数据源，选择其一并统一使用：

方案一（DNAGold 为权威数据源）：
// EconomyEngine.getLockedBalance 代理到 DNAGold：
function getLockedBalance(uint256 agentId) external view returns (uint256) {
    return dnagold.lockedBalance(agentId);  // 代理读取 DNAGold
}

方案二（EconomyEngine 为权威数据源）：
// DNAGold.addLocked 代理到 EconomyEngine，DNAGold 不独立维护 lockedBalance
// AgentBridge 直接读取 EconomyEngine（现有架构）

无论选择哪种方案，必须确保只有一个合约是 lockedBalance 的写入权威，消除双写导致的不一致风险。

问题描述
DNAGold 作为游戏代币，若某地址通过漏洞获得了大量非法 DNAGOLD，目前没有任何机制可以冻结该地址或标记为黑名单，无法阻止其将非法所得转出或抛售。
这是一个「是否需要中心化干预能力」的设计哲学问题。去中心化纯粹主义者会认为 freeze 机制违背 Web3 精神；而实际游戏运营中，应对漏洞攻击的紧急响应能力是必要的。
影响分析
这取决于项目方对去中心化程度的判断，无强制要求。若项目方认为需要此能力，可接受对去中心化属性的一定牺牲：
修复建议
若决定添加 freeze 能力：
mapping(address => bool) public frozen;
function freeze(address addr) external onlyOwner { frozen[addr] = true; }

// 在 _beforeTokenTransfer 中检查：
function _beforeTokenTransfer(address from, address to, uint256 amount) internal override {
    require(!frozen[from], "Sender is frozen");
    super._beforeTokenTransfer(from, to, amount);
}
注意：freeze 机制需要透明披露，并建议由多签控制，防止 freeze 权力被滥用。若不添加，建议在代币白皮书中明确说明无 freeze 能力。

问题描述
DNAGold 没有 burn 或 burnFrom 函数。随着游戏运行，通过 mintFree 持续铸造的新代币（每次 Gather 的 hostShare 部分）会使流通量持续增加。减半机制（EconomyEngine）在一定程度上控制了增发速度，但没有任何通过销毁代币来抵消的机制。
若游戏设计了消耗 DNAGOLD 的玩法（如升级、购买装备、繁殖费用等），这些消耗应通过 burn 将 DNAGOLD 从流通中永久移除，形成完整的通缩闭环。
影响分析
若游戏规则中有消耗 DNAGOLD 的机制，缺少 burn 函数意味着这些消耗要么被发送到死地址（无法追踪），要么无法实现。这会使通胀控制机制不完整。
修复建议
添加标准 OpenZeppelin ERC20Burnable 扩展，或手动实现：
import "@openzeppelin/contracts/token/ERC20/extensions/ERC20Burnable.sol";
contract DNAGold is ERC20, ERC20Burnable, Ownable { ... }

burn 函数允许持有者主动销毁自己的代币，构成经济循环的通缩端，与 mintFree 的通胀端形成平衡。建议在游戏设计层面明确哪些操作会触发 burn（如繁殖费用、特殊能力激活等）。

5. 总结与修复优先级
5.1 整体评估
DNAGold 继承 OpenZeppelin ERC20，基础实现安全可靠。最关键的问题是 P-1（mintFree 无总量上限）和 P-2（锁仓函数权限确认）——这两项是整个系统的最后防线。P-1 的修复只需一行代码，但效果是将「DNAGOLD 是否有真实总量上限」从模糊变为明确保证，对代币经济可信度影响深远。

CodeDNA 智能合约安全审计报告
 

合约 #10 — ReproductionEngine (繁殖引擎)
BNB Smart Chain 主网  Agent 繁殖逻辑  亲属检查  NFT 铸造调用


1. 审计概述
ReproductionEngine 负责 CodeDNA 中 Agent 的繁殖逻辑，是 BehaviorEngine 调用 reproduce() 时的核心后端合约。它负责验证两个父母 Agent 的合法性（存活、性别、亲属关系、繁殖冷却），并在所有条件满足后触发 GenesisCore.mintAgent 铸造新 Agent NFT，同时通知 FamilyTracker 和 EconomyEngine 更新相关状态。

⚠️ 架构说明： BehaviorEngine 中原有的 reproduce() 逻辑可能已经包含了亲属检查和 mint 调用，ReproductionEngine 作为独立合约可能是将繁殖逻辑从 BehaviorEngine 中拆分出来的专用模块。需确认两者的实际分工边界，避免逻辑重复或遗漏。


2. 链上实时状态

3. 设计文档一致性核查

4. 安全风险详情
本次审计共发现 1个高风险、3个中风险、3个低风险 问题。P-1（亲属检查深度不足）和 P-3（同宿主绕过）是繁殖系统经济安全的核心风险。


问题详情
问题描述
ReproductionEngine 通过 FamilyTracker.isDirectRelative(fatherId, motherId) 检查亲属关系，但根据 FamilyTracker 审计（#4，P-4），isDirectRelative 只检查直接父子关系（2层），祖孙关系（3层）、堂兄弟（同祖父母的不同子女）均无法被检测到。
在游戏中，玩家可以有意识地进行近亲繁殖策略：让高属性祖父与孙女繁殖，或让同一父母的兄妹繁殖，快速积累高属性 Agent，破坏游戏的属性平衡设计。
相关代码
// FamilyTracker.isDirectRelative 只检查2层：
function isDirectRelative(uint256 aId, uint256 bId) external view returns (bool) {
    // 仅检查 a是b的父母 或 b是a的父母
    return parents[bId][0]==aId || parents[bId][1]==aId ||
           parents[aId][0]==bId || parents[aId][1]==bId;
    // ❌ 未检查：祖父/祖母关系（3层）
    // ❌ 未检查：同父母兄弟姐妹（共同父母）
}
影响分析
攻击者可设计繁殖路径：A+B→C，A+D→E，然后 C+E 繁殖（同父异母兄妹）。isDirectRelative 检查 C 和 E 的直系关系时，因为 C 和 E 不是彼此的父母，返回 false，允许繁殖。通过多代近亲繁殖可迅速培育全高属性 Agent，对游戏经济平衡造成破坏。
修复建议
在 FamilyTracker 层添加 familyRoot 近亲检查：
function isCloseRelative(uint256 aId, uint256 bId) external view returns (bool) {
    // 检查是否共享同一 familyRoot（3代内）
    bytes32 rootA = familyRoot[aId];
    bytes32 rootB = familyRoot[bId];
    if (rootA != bytes32(0) && rootA == rootB) return true;
    // 检查同父母（兄弟姐妹）
    if (parents[aId][0] != 0 && (parents[aId][0]==parents[bId][0] || parents[aId][0]==parents[bId][1])) return true;
    if (parents[aId][1] != 0 && (parents[aId][1]==parents[bId][0] || parents[aId][1]==parents[bId][1])) return true;
    return false;
}
ReproductionEngine 中将 isDirectRelative 替换为 isCloseRelative 调用。

问题描述
ReproductionEngine 对每个 Agent 的繁殖限制可能仅通过 reproduceCount <= MAX_REPRODUCE_COUNT 来控制，没有基于区块数的冷却时间（如 lastReproduceBlock + REPRODUCE_COOLDOWN）。这意味着在单次交易或连续区块内，玩家可以让 Agent 快速完成所有允许的繁殖次数，快速产出大量后代。
这与 gather 无冷却（BehaviorEngine P-3）的问题性质相同：次数限制可以在短时间内被耗尽，而不是按设计意图分散在游戏周期内。
相关代码
uint256 public constant MAX_REPRODUCE_COUNT = 5;  // 生命周期内最多繁殖5次

function reproduce(uint256 fatherId, uint256 motherId) external {
    // ✅ 次数检查存在
    require(genesisCore.reproduceCount(fatherId) < MAX_REPRODUCE_COUNT, "Father exhausted");
    require(genesisCore.reproduceCount(motherId) < MAX_REPRODUCE_COUNT, "Mother exhausted");
    // ❌ 无区块冷却：lastReproduceBlock 未检查
    // 可在 5 个连续区块内完成全部 5 次繁殖
    ...
}
影响分析
玩家可以通过脚本在最短时间内完成 Agent 的全部繁殖配额，快速产出最多后代，不受设计意图中「繁殖需要时间」的约束。
修复建议
添加基于区块的繁殖冷却（约1天=28800区块）：
mapping(uint256 => uint256) public lastReproduceBlock;
uint256 public constant REPRODUCE_COOLDOWN = 28800;

require(block.number >= lastReproduceBlock[fatherId] + REPRODUCE_COOLDOWN, "Father cooldown");
require(block.number >= lastReproduceBlock[motherId] + REPRODUCE_COOLDOWN, "Mother cooldown");
// 繁殖成功后更新：
lastReproduceBlock[fatherId] = block.number;
lastReproduceBlock[motherId] = block.number;

问题描述
ReproductionEngine 的 reproduce() 要求调用者是 msg.sender，但若检查逻辑为 ownerOf(fatherId) == msg.sender && ownerOf(motherId) == msg.sender，则不允许不同宿主之间的 Agent 繁殖。若设计允许跨宿主繁殖（PvP 繁殖模式），则需要双方签名或授权机制，否则 A 宿主的 Agent 可能被 B 宿主的 Agent「强制繁殖」。
若仅检查 ownerOf(fatherId) == msg.sender，则 B 宿主可以用 B 的 fatherId + A 的 motherId（只要 motherId 在同一地块），在 A 不知情的情况下消耗 A 的 Agent 繁殖次数。
相关代码
// 若只检查 father 的宿主：
require(genesisCore.ownerOf(fatherId) == msg.sender, "Not father owner");
// ❌ 未检查 mother 的宿主
// B 可用自己的 fatherId + A 的 motherId 调用
// 消耗 A 的 mother 繁殖次数，A 不知情
影响分析
攻击者可消耗其他玩家 Agent 的繁殖配额，使其无法完成自己计划的繁殖。虽然攻击者也消耗自己 Agent 的配额，属于零和攻击，但对受害者造成实际资产价值损失（繁殖机会是有限资源）。
修复建议
强制要求两个 Agent 归属同一宿主（若设计为单宿主繁殖）：
require(genesisCore.ownerOf(fatherId) == msg.sender, "Not father owner");
require(genesisCore.ownerOf(motherId) == msg.sender, "Not mother owner");

若设计支持跨宿主繁殖，需引入双方授权机制：
mapping(uint256 => mapping(uint256 => bool)) public reproduceApproval;
// mother 宿主提前授权 father 与其繁殖
function approveReproduce(uint256 motherId, uint256 fatherId) external {
    require(ownerOf(motherId) == msg.sender);
    reproduceApproval[motherId][fatherId] = true;
}

问题描述
设计文档中繁殖应需要消耗一定 DNAGOLD（繁殖费用），这构成 DNAGOLD 的消耗端（通缩），与 Gather 奖励的铸造端（通胀）形成经济闭环。需确认 ReproductionEngine 中是否有 dnagold.transferFrom(msg.sender, treasury/burn, reproduceFee) 的调用。
若繁殖费用为零（未实现），则玩家可以无成本地持续繁殖（受次数限制），新 Agent 的铸造完全免费，会加速 Agent 总数增长，稀释每个 Agent 的 Gather 产出（稀释因子上升），影响所有玩家的收益。
相关代码
// 需确认是否存在：
uint256 public reproduceFee = 100 * 1e18;  // 100 DNAGOLD
dnagold.transferFrom(msg.sender, address(0), reproduceFee);  // burn
// 或
dnagold.transferFrom(msg.sender, treasury, reproduceFee);  // 入库
// ❓ 若此调用缺失，繁殖完全免费
影响分析
若繁殖费用缺失，玩家在 Agent 生命周期内可免费获得最多 MAX_REPRODUCE_COUNT 个后代，加速 Agent 总数膨胀，破坏 EconomyEngine 的稀释机制设计。
修复建议
确认并补充繁殖费用收取逻辑：
uint256 public reproduceFee;  // 由 owner 设置
if (reproduceFee > 0) {
    require(dnagold.transferFrom(msg.sender, burnAddress, reproduceFee), "Fee failed");
}
建议将繁殖费用设为 burn（销毁），与 DNAGold P-6 的通缩机制补充配合，形成完整的代币销毁场景。

问题描述
若新生 Agent 的属性为父母属性的均值（或加权平均），玩家可以通过「属性配对」策略：持续选择高属性 Agent 繁殖，逐代提升后代属性，最终培育出远超初始属性上限的 Agent（属性最大值理论上趋近于父母最大值的上界，而非固定上限）。
这是否符合设计意图取决于游戏设计者的考量：属性为遗传特征可能是有意为之的进化玩法设计。
影响分析
若属性遗传确实是设计意图，当前实现正确，但需在游戏文档中说明「属性可通过繁殖提升」的上限规则。
若需要限制属性膨胀，可引入属性衰退因子：newAttr = (fAttr + mAttr) / 2 * 0.95 + random()，确保属性不会无限提升。
修复建议
属性遗传机制是设计决策，建议在游戏白皮书中明确说明属性遗传规则，供玩家理解游戏经济平衡。

问题描述
BehaviorEngine 的 reproduce() 函数（审计 #7）已经包含了部分繁殖逻辑（亲属检查、mintAgent 调用、registerBirth、incrementBorn）。ReproductionEngine 作为独立合约的引入，需要明确其与 BehaviorEngine 的职责划分，否则会出现：双重亲属检查（浪费 gas）；BehaviorEngine 绕过 ReproductionEngine 直接调用 GenesisCore.mintAgent（逻辑不一致）；ReproductionEngine 的冷却管理与 BehaviorEngine 的状态更新不同步。
影响分析
若 BehaviorEngine 直接调用 GenesisCore.mintAgent 而不经过 ReproductionEngine，则 ReproductionEngine 的安全检查完全被绕过，P-1 至 P-4 的所有问题都无法起到防护作用。
修复建议
明确架构边界：BehaviorEngine.reproduce() 应该仅作为入口，所有繁殖逻辑（验证、铸造、注册）统一由 ReproductionEngine 处理。BehaviorEngine 不应直接调用 GenesisCore.mintAgent。建议在代码注释和文档中明确标注调用链：BehaviorEngine.reproduce() → ReproductionEngine.executeReproduce() → GenesisCore.mintAgent()。

问题描述
FamilyTracker.removeAgent（审计 #4，P-3）不清除 parents[]、registered、familyRoot 等状态。若死亡 Agent 的亲属关系数据仍保留，ReproductionEngine 在调用 isDirectRelative 时可能对已死亡的亲属关系做出错误判断（如允许与已死亡父母重新配对）或导致亲属数据不一致。
影响分析
若死亡 Agent 的 parents 记录未清除，其后代在进行亲属检查时仍能正确识别与死亡 Agent 的亲属关系，不会产生直接安全漏洞。但数据残留会导致 FamilyTracker 中存在大量幽灵数据，影响链上查询准确性。
修复建议
与 FamilyTracker P-3 修复同步处理：removeAgent 时清除 parents[id]、registered[id]、familyRoot[id]，确保死亡后亲属数据一致。参见 FamilyTracker 审计 #4 的修复建议。

CodeDNA 智能合约安全审计报告
 

合约 #11 — NFTSale（创世 Agent 铸造销售）
BNB Smart Chain 主网  |  用户支付 BNB 铸造 Genesis Agent NFT  |  真实 BNB 资金流转


1. 审计概述
NFTSale 是 CodeDNA 的创世 Agent 一级销售合约，是整个生态最先与真实资金接触的入口。玩家通过调用 mintGenesis 支付 0.1 BNB，合约调用 GenesisCore.mintFor 为玩家铸造 Genesis Agent NFT，同时向预设收款地址分发 0.02 BNB（20%收益分成）。合约还持有 50,000 DNAGOLD 作为购买奖励池。

⚠️ 高优先级警示： NFTSale 直接持有并处理真实 BNB，是全系统唯一的资金入口合约。任何与 BNB 处理相关的漏洞（重入、超额收款、不可提款）都会造成真实资产损失，审计优先级高于其他纯逻辑合约。


2. 链上实时状态

3. 设计文档一致性核查

4. 安全风险详情
本次审计共发现 2个高风险、3个中风险、2个低风险 问题。P-3（重入攻击）和 P-4（BNB 提款锁定）是直接威胁真实资金安全的核心风险，需立即处理。


问题详情
问题描述
mintGenesis 函数需要验证用户支付的 BNB 数量是否等于铸造价格。价格的来源有两种可能：①从 GenesisCore.getGenesisPrice() 动态读取（灵活但依赖外部调用）；②在 NFTSale 内部硬编码（固定但无法调整）。
若价格硬编码在 NFTSale 中（如 uint256 public constant PRICE = 0.1 ether），当市场价格需要调整时，必须重新部署 NFTSale 合约，期间用户无法铸造，且旧合约中已收取的 0.4 BNB 需要手动迁移，存在操作风险。
此外，若价格在 GenesisCore 和 NFTSale 中各自独立维护，两者可能不同步，造成用户按错误价格支付。
相关代码
// 方案A（推荐）：动态读取 GenesisCore 价格
uint256 price = genesisCore.getGenesisPrice();
require(msg.value >= price, "Insufficient BNB");

// 方案B（风险）：NFTSale 内部硬编码
uint256 public mintPrice = 0.1 ether;  // 需 owner 手动更新
require(msg.value >= mintPrice, "Insufficient BNB");
影响分析
若 NFTSale 硬编码价格后 GenesisCore 价格更新，两合约价格不一致。玩家可能以旧价格铸造，合约无法按新价格收费，导致收益损失；或价格过高导致用户无法铸造，影响销售进度。
修复建议
统一价格来源：NFTSale 的 mintPrice 从 GenesisCore.getGenesisPrice() 读取，不独立维护价格字段。若需本地缓存，添加 syncPrice() 函数由 owner 手动同步，并 emit PriceSynced 事件。

问题描述
mintGenesis 函数为 payable，若校验逻辑使用 require(msg.value >= price) 而非 require(msg.value == price)，用户多付的 BNB 会被留存在合约中，既没有退款给用户，又没有计入合约方收益（因为收益分成只按固定 0.02 BNB 计算）。
链上数据验证：5 笔铸造交易的 value 分别为 0.05088326、0.05325、0.05、0.05764533、0.05 BNB（单位从较低精度看，但注意实际收益分成每次固定 0.02 BNB），与预期 0.1 BNB 不符。若价格确为 0.1 BNB，则这些值显示用户可能以不同价格铸造，需进一步核查。
无论如何，缺少超额退款逻辑是高风险设计缺陷。
相关代码
// 当前（推测）：只检查最低金额
require(msg.value >= price, "Insufficient BNB");
// ❌ 多付的 BNB 被锁在合约

// 正确实现：精确匹配或退款
require(msg.value >= price, "Insufficient BNB");
uint256 refund = msg.value - price;
if (refund > 0) {
    (bool sent,) = payable(msg.sender).call{value: refund}("");
    require(sent, "Refund failed");
}
影响分析
用户若多支付 BNB（例如多付 0.01 BNB），该部分直接被合约吸收，用户无法追回。5 笔交易若均存在超付，总计可能有 0.02-0.05 BNB 的用户资金被意外锁定。
修复建议
修改铸造逻辑为精确金额校验或自动退款：方案 A（严格）：require(msg.value == price, "Exact BNB required")；方案 B（友好）：超额部分自动退回。推荐方案 B，同时在 mintGenesis 前加 nonReentrant 防止退款被重入利用。

问题描述
mintGenesis 内部在铸造完成后，通过 payable(treasury).call{value: 0.02 ether}("") 将 20% 收益转入收款地址。若收款地址（0xC2fDF351...b219）是一个合约且其 receive() 函数重新调用 mintGenesis，则在 minted NFT 数量更新前，攻击者可以多次铸造，绕过 MAX_GENESIS 上限。
链上观察：0xC2fDF351...b219 每次都收到 0.02 BNB，该地址是否为合约需确认。若为 EOA 地址（外部账户），重入风险较低；若为合约地址，则需要关注其 receive() 实现。
相关代码
function mintGenesis() external payable {
    require(genesisCore.totalGenesisCount() < genesisCore.MAX_GENESIS());
    genesisCore.mintFor(msg.sender, ...);  // 铸造 NFT
    // ❌ 铸造后立即进行外部调用
    (bool sent,) = payable(treasury).call{value: 0.02 ether}("");
    require(sent, "Treasury transfer failed");
    // 若 treasury 是合约且重入调用，totalGenesisCount 可能未更新
}
影响分析
若收款地址是含恶意代码的合约，可在铸造后重入 mintGenesis，绕过总量检查，连续铸造超出 MAX_GENESIS 上限的 Agent，破坏创世 Agent 的稀缺性。
修复建议
添加 nonReentrant 修饰符（OpenZeppelin ReentrancyGuard）；或遵循 CEI 模式（先更新状态，再进行外部调用）；或将收益分成改为「记录待提取」模式，由 treasury 主动调用 claimRevenue() 提取，避免铸造流程中发起外部调用。

问题描述
每笔 0.1 BNB 铸造中，0.02 BNB 转至 treasury，剩余 0.08 BNB 留在合约。当前合约余额 0.4 BNB（5 笔 × 0.08 BNB）。需确认合约是否存在安全的 withdraw() 函数供 owner 提取，以及 owner 地址是否为多签钱包。
风险一：若无 withdraw 函数，0.4 BNB 永久锁定合约，无法提取（合约不可自毁）。风险二：若 withdraw 函数无 onlyOwner 保护，任意地址可提取所有 BNB。风险三：若 owner 是单签 EOA，私钥泄露即全额损失。
相关代码
// 需确认以下函数存在：
function withdraw() external onlyOwner {
    uint256 balance = address(this).balance;
    (bool sent,) = payable(owner()).call{value: balance}("");
    require(sent, "Withdraw failed");
    emit Withdrawn(owner(), balance);
}

// 如果只有 receive() 无 withdraw()，BNB 被永久锁定
影响分析
若 withdraw 函数不存在或存在权限漏洞，当前 0.4 BNB 及后续所有铸造收入将面临锁定或被盗风险。随着销售量增加，合约内 BNB 余额将持续增长，风险敞口随之扩大。
修复建议
确认 withdraw 函数存在且为 onlyOwner；owner 迁移至多签钱包（Gnosis Safe 2/3），防止单点私钥风险；考虑添加 withdrawTo(address) 支持灵活提款地址；提款时 emit Withdrawn 事件便于监控。

问题描述
合约当前持有 50,000 DNAGOLD，这批代币可能用于：①作为铸造奖励，每次 mintGenesis 向买家分发固定数量 DNAGOLD；②作为早期流动性奖励，在销售结束后由 owner 手动分发；③误转入，非设计功能。
若①（分发逻辑存在但有 bug），可能发生超额分发（50,000 DNAGOLD 被少数玩家领走）或欠额分发（DNAGOLD 永久锁定）。若③（误转入），需要 owner 通过 rescueToken 函数取回，否则 50,000 DNAGOLD 永久锁定。
相关代码
// 可能的实现（需确认）：
uint256 public dnagoldBonus = 1000 * 1e18;  // 每次铸造奖励 1000 DNAGOLD

function mintGenesis() external payable {
    // ...价格检查...
    genesisCore.mintFor(msg.sender, newTokenId);
    dnagold.transfer(msg.sender, dnagoldBonus);  // 分发奖励
    // ❓ 若 dnagold 余额耗尽但仍有人铸造，transfer 失败导致整个铸造失败
}
影响分析
若 DNAGOLD 分发逻辑存在缺陷，可能导致早期购买者无法获得应有奖励，或 50,000 DNAGOLD（约占总供应量 0.005%）永久锁定在无法访问的合约中。
修复建议
确认 DNAGOLD 分发设计：若为铸造奖励，添加余额检查 require(dnagold.balanceOf(this) >= bonus)，当余额不足时停止铸造或跳过奖励；添加 rescueToken(address token, uint256 amount) onlyOwner 函数，允许 owner 提取意外锁定的 ERC20 代币。

问题描述
当前 mintGenesis 仅受 MAX_GENESIS 总量限制，无单地址铸造数量限制。单一地址可通过多笔交易铸造所有剩余 Genesis Agent（只要支付足够 BNB），垄断稀缺资源，破坏游戏公平性。
创世 Agent 是稀缺 NFT，设计意图通常是供多名玩家广泛持有，而非被少数地址垄断。
影响分析
若 MAX_GENESIS 较小（例如 100 个），单人垄断风险极高。若 MAX_GENESIS 较大，风险相对降低。
修复建议
添加每钱包铸造上限：
uint256 public maxPerWallet = 5;
mapping(address => uint256) public mintedCount;
require(mintedCount[msg.sender] + 1 <= maxPerWallet, "Wallet limit reached");
mintedCount[msg.sender]++;

问题描述
NFTSale 可能缺少 Pausable 功能或 saleActive 开关。若铸造过程中发现重入漏洞或价格异常，无法立即暂停销售，攻击者可在修复期间持续利用漏洞。
相关代码
// 建议添加：
bool public saleActive = true;
modifier whenSaleActive() { require(saleActive, "Sale paused"); _; }
function setSaleActive(bool active) external onlyOwner { saleActive = active; }

function mintGenesis() external payable whenSaleActive { ... }
影响分析
若漏洞发生时无法暂停，开发团队无法有效阻止损失扩大。
修复建议
继承 OpenZeppelin Pausable 合约，铸造函数加 whenNotPaused 修饰符；pause() 权限授予多签钱包；在销售结束后调用 pause() 永久停止铸造入口。

5. 总结与修复优先级
⚠️ 特别警示： NFTSale 是全系统唯一持有真实 BNB 的合约，当前余额 0.4 BNB（约 $261）且随销售增长。P-2 和 P-4 需立即验证，防止用户资金损失和合约资金锁定。

CodeDNA 智能合约安全审计报告
 

合约 #12 — NFTMarket（Agent 二级交易市场）
BNB Smart Chain 主网  |  Agent NFT 挂单 / 取消 / 购买  |  BNB 资金撮合


1. 审计概述
NFTMarket 是 CodeDNA 生态的 Agent NFT 二级交易市场，支持玩家将 Agent NFT 挂单出售，其他玩家支付 BNB 购买。已观测到的方法为 listAgent（挂单）和 cancelListing（取消挂单），完整购买流程还包含 buyAgent 方法。

NFTMarket 是全系统第二个与真实 BNB 资金直接交互的合约，也是 Agent NFT 流转的核心中转站。NFT 市场类合约的典型高危风险包括：重入攻击（购买时 BNB 转账触发回调）、幽灵挂单（Agent 死亡 / 转移后挂单未清除）、超额支付无退款以及平台手续费管理不当。


2. 链上实时状态

3. 设计文档一致性核查

4. 安全风险详情
本次审计共发现 2个高风险、3个中风险、3个低风险 问题。P-2（购买重入）与 P-1（幽灵挂单）是 NFT 市场类合约最高频的被攻击漏洞，需在开放购买功能前完成修复。


问题详情
问题描述
NFT 市场有两种挂单模式：① Escrow 模式（listAgent 调用时 NFT 立即转入 NFTMarket 合约托管）；② Approve 模式（listAgent 仅授权合约，NFT 仍留在卖家手中）。
若 NFTMarket 使用 Approve 模式，卖家在挂单后可以随时通过 GenesisCore.transferFrom 将 Agent 转移至其他地址，使 approve 失效。此时若有买家触发 buyAgent，合约会尝试 transferFrom(seller, buyer, tokenId)，但由于 NFT 已转走，此调用必然失败。
关键风险点：若购买合约在 NFT 转移失败后未正确回滚 BNB 转账，买家将损失已支付的 BNB 却得不到 NFT。即使合约能正确 revert，Approve 模式也导致大量「无效挂单」存在于市场，严重损害用户体验与市场公信力。
相关代码
// ❌ Approve 模式（有风险）
function listAgent(uint256 tokenId, uint256 price) external {
    require(genesisCore.ownerOf(tokenId) == msg.sender, "Not owner");
    genesisCore.approve(address(this), tokenId);   // 仅授权，NFT 留在卖家
    listings[tokenId] = Listing(msg.sender, price, true);
}

// ✅ Escrow 模式（推荐）
function listAgent(uint256 tokenId, uint256 price) external {
    require(genesisCore.ownerOf(tokenId) == msg.sender, "Not owner");
    genesisCore.transferFrom(msg.sender, address(this), tokenId);  // NFT 进托管
    listings[tokenId] = Listing(msg.sender, price, true);
}
影响分析
在 Approve 模式下，卖家可以反复「挂单-转走-再上架」实施钓鱼攻击，诱导买家触发失败的购买交易，消耗 gas 的同时破坏市场公信力；极端情况下，若代码存在 BNB 先转后 NFT 失败的顺序缺陷，可造成买家 BNB 永久损失。
修复建议
将挂单改为 Escrow 模式：listAgent 时通过 transferFrom 将 NFT 转入市场合约；cancelListing 时将 NFT 归还卖家；buyAgent 时从合约直接转给买家。Escrow 模式消除了挂单期间所有权变化的风险。

问题描述
buyAgent 的核心执行顺序是：① 校验买家支付金额；② 将 NFT 转移给买家；③ 向卖家转账 BNB（扣手续费）。若第③步的 BNB 转账使用 call{value}，且转账发生在删除挂单记录之前，恶意卖家合约可以在 receive() 中重新调用 buyAgent，在 listings[tokenId] 尚未被 delete 的情况下再次触发购买流程，实现重复消耗买家 BNB 或转走多份 NFT 的攻击。
这是 NFT 市场合约最经典的重入漏洞场景，历史上 Nifty Gateway、多个 DeFi NFT 市场均因此类漏洞遭受损失。
相关代码
// ❌ 错误顺序（重入漏洞）
function buyAgent(uint256 tokenId) external payable {
    Listing storage l = listings[tokenId];
    require(l.active, "Not listed");
    require(msg.value >= l.price, "Insufficient");
    // 先转 BNB → 触发重入 → listings 未清除 → 再次购买
    payable(l.seller).call{value: l.price}("");
    delete listings[tokenId];   // ❌ 太晚了
    genesisCore.transferFrom(address(this), msg.sender, tokenId);
}

// ✅ 正确：CEI 模式 + nonReentrant
function buyAgent(uint256 tokenId) external payable nonReentrant {
    Listing memory l = listings[tokenId];   // memory 快照
    require(l.active, "Not listed");
    require(msg.value >= l.price, "Insufficient");
    delete listings[tokenId];               // 先清除状态
    genesisCore.safeTransferFrom(address(this), msg.sender, tokenId);
    uint256 fee = l.price * platformFee / 10000;
    payable(l.seller).call{value: l.price - fee}("");  // 最后转 BNB
    if (msg.value > l.price) payable(msg.sender).call{value: msg.value - l.price}("");
}
影响分析
成功重入的攻击者（恶意卖家合约）可以用单笔购买多次触发 listings 读取，在一次 buyAgent 调用链中重复骗取买家 BNB，或者多次转移同一 NFT。整个过程在单个区块内完成，无法通过链上监控及时阻止。
修复建议
① 强制添加 nonReentrant 修饰符（OpenZeppelin ReentrancyGuard）；② 严格遵循 CEI 模式：delete listings 必须在任何外部调用（NFT 转移、BNB 转账）之前执行；③ 建议将卖家 BNB 改为「记录余额-主动提款」模式（Pull Payment），彻底消除购买流程中的外部调用重入风险。

问题描述
当 DeathEngine 判定某 Agent 死亡时，它会更新 isDead 状态并可能触发 Agent NFT 的转移或 burn。但 NFTMarket 的 listings 映射不会自动清除已死亡 Agent 的挂单记录，导致市场上出现「幽灵挂单」——挂单看起来有效，但对应的 Agent 已失去全部游戏功能。
买家若购买已死亡的 Agent，虽然技术上完成了 NFT 转移，但获得的是一个无法参与任何游戏活动的无价值资产。若市场 UI 不主动过滤死亡 Agent 的挂单，玩家将在不知情的情况下遭受损失。
与 DeathEngine 审计（#6，P-2：WorldMap 未同步）、FamilyTracker 审计（#4，P-3：removeAgent 不清理状态）是同类的跨合约状态不同步问题。
相关代码
// DeathEngine 执行死亡后，NFTMarket listings 未清除

    worldMap.removeAgent(agentId);

    // listings[agentId] 仍然 active = true
}

// buyAgent 时若不检查存活状态，可购买死亡 Agent
function buyAgent(uint256 tokenId) external payable {
    // ❌ 缺少：require(!deathEngine.isDead(tokenId), "Agent is dead");
}
影响分析
已死亡的 Agent NFT 在游戏中价值为零，但挂单价格可能接近正常 Agent 的市场价。不知情的买家支付真实 BNB 后只能持有一个「装饰性」NFT，无法参与任何 CodeDNA 游戏活动，造成实质性的 BNB 损失。
修复建议
双重防护方案：① 在 buyAgent 中添加存活检查：require(!deathEngine.isDead(tokenId), "Agent is dead")，若已死亡则 revert 并退还 BNB；② DeathEngine 执行死亡逻辑时主动调用 nftMarket.cancelListing(agentId)（需 DeathEngine 持有相应权限），确保死亡即时撤单；③ 前端过滤死亡 Agent 的挂单展示。

问题描述
NFTMarket 的 platformFee 决定每笔交易中平台抽成的比例。若 setPlatformFee 函数仅有 onlyOwner 保护且无手续费上限（MAX_FEE）限制，合约部署者可以在任何时刻将手续费设为 100%，使下一笔购买的全部 BNB 进入 feeRecipient，而卖家分文未得。
即使团队无恶意，此设计也存在因 owner 私钥被盗而导致手续费被攻击者篡改的风险。另外，feeRecipient 若为单签 EOA，私钥泄露即导致所有历史累积手续费和未来手续费被盗。
相关代码
// ❌ 无上限、可即时修改的手续费
uint256 public platformFee = 250;   // 初始 2.5%（基数 10000）
address public feeRecipient;

function setPlatformFee(uint256 newFee) external onlyOwner {
    platformFee = newFee;   // ❌ 可设为 10000（100%），无上限检查
}

// ✅ 应有上限保护
uint256 public constant MAX_FEE = 1000;   // 最高 10%
function setPlatformFee(uint256 newFee) external onlyOwner {
    require(newFee <= MAX_FEE, "Fee too high");
    platformFee = newFee;
    emit PlatformFeeUpdated(newFee);
}
影响分析
若攻击者获取 owner 私钥，可将 platformFee 设为 10000（100%），此后每笔购买的全部 BNB 进入 feeRecipient，卖家颗粒无收，买家付出真实 BNB 却得不到应有价值。整个 NFT 市场经济将被瞬间摧毁。
修复建议
① 设置 MAX_FEE 常量（建议 ≤10%）并在 setPlatformFee 中强制检查；② 将 feeRecipient 迁移至多签钱包（Gnosis Safe 2/3）；③ 考虑为 setPlatformFee 添加 TimeLock（如 48 小时），让社区有足够时间发现并响应恶意修改。

问题描述
buyAgent 作为 payable 函数，若校验逻辑为 require(msg.value >= l.price) 而非精确匹配，买家多支付的 BNB 将留在合约（若合约无 withdraw）或直接进入系统（若合约即时流转）。无论哪种情况，买家均无法追回多付的 BNB。
此问题与 NFTSale P-2 完全相同，是 payable 函数的通用设计缺陷，在二级市场场景中因用户可能手动输入价格而更易发生（用户输入 0.15 BNB 购买标价 0.1 BNB 的 Agent）。
相关代码
// ❌ 无退款：多付的 BNB 消失
require(msg.value >= l.price, "Insufficient BNB");
// msg.value = 0.15 BNB，l.price = 0.1 BNB
// 多出的 0.05 BNB 被永久锁定或进入手续费池

// ✅ 有退款
require(msg.value >= l.price, "Insufficient BNB");
uint256 excess = msg.value - l.price;
if (excess > 0) {
    (bool refunded,) = payable(msg.sender).call{value: excess}("");
    require(refunded, "Refund failed");
}
影响分析
每笔交易若存在 0.01-0.05 BNB 的超额支付，随着市场交易量增长，累计锁定的 BNB 可能达到数十至数百美元，对用户造成不可忽视的资金损失。
修复建议
在 buyAgent 末尾添加自动退款逻辑（在 CEI 模式中，退款调用置于最后）；或改为严格精确金额校验 require(msg.value == l.price)；推荐前者以提升用户体验。退款调用同样需要在 nonReentrant 保护下执行。

问题描述
listAgent 如果未 require(price > 0)，卖家可以意外地以 0 BNB 挂单。链上机器人（MEV bot）实时监控挂单事件，一旦检测到 price == 0 的挂单，会在同一区块内以 0 BNB 购买，卖家颗粒无收。即使合约本身不损失资金，此类事故会严重损害用户对市场的信任。
相关代码
// ❌ 缺少零价格检查
function listAgent(uint256 tokenId, uint256 price) external {
    // 未检查 price > 0
    listings[tokenId] = Listing(msg.sender, price, true);
}

// ✅ 加入最低价格检查
uint256 public constant MIN_PRICE = 0.001 ether;   // 最低 0.001 BNB
require(price >= MIN_PRICE, "Price too low");
影响分析
卖家因误操作以 0 BNB 挂单后，在同区块即被机器人购走，NFT 永久损失且无法追回。
修复建议
添加 MIN_PRICE 常量检查（建议 ≥ 0.001 BNB）；前端 UI 对极低价格挂单展示警告提示，增加确认步骤。

问题描述
若 NFTMarket 不支持 Pausable（全局暂停），当出现重入漏洞或价格操纵等紧急情况时，开发团队无法立即冻结市场，攻击者可在修复部署前持续利用漏洞。鉴于 P-1、P-2 等高危漏洞的存在，暂停机制是必要的安全底线。
相关代码

bool public marketActive = true;

    require(marketActive, "Market paused");
    _;

// listAgent、buyAgent 均加 whenMarketActive 修饰符
影响分析
在高危漏洞被发现到修复部署的窗口期内（通常数小时至数天），若无暂停机制，损失将持续扩大。
修复建议
继承 OpenZeppelin Pausable；pause() 权限授予多签钱包；buyAgent、listAgent 加 whenNotPaused；在正式上线前进行一次暂停-恢复演练确保机制正常工作。

问题描述
当 Agent 在 WorldMap 上处于活跃状态（agentOnMap = true）时，若玩家将其挂单至 NFTMarket，Agent 可能仍出现在地图上进行 Gather、Reproduce 等操作，同时其 NFT 处于市场挂单状态。若 buyAgent 成功，新宿主接管 NFT，但原宿主的 Agent 仍留在地图上（因为 WorldMap.agentLocation 以 agentId 为 key，与所有权无关）。
这会导致：① 卖家在挂单期间可以继续操作 Agent，获取 Gather 收益，相当于「出售后继续使用」；② 购买者获得 NFT 后，Agent 仍在原宿主控制的地图位置，无法立即上手游玩。
影响分析
虽然不直接造成资金损失，但此类状态不同步破坏游戏公平性，可能被玩家滥用（大量挂单低价 Agent 以套取 Gather 收益同时出售）。
修复建议
在 listAgent 时检查 Agent 是否在地图上，若在则强制调用 worldMap.removeAgent(tokenId) 后再完成挂单；buyAgent 完成后通知 WorldMap 更新所有权信息。在 GameContract / BehaviorEngine 中，增加对 Agent 是否挂单中的检查，禁止挂单中的 Agent 执行 Gather / Reproduce 操作。

5. 总结与修复优先级
⚠️ 警示： 当前 buyAgent 尚无链上交易记录，市场处于测试/早期阶段。P-1 和 P-2 必须在 buyAgent 被正式使用前完成修复，否则一旦有真实 BNB 购买发生，攻击窗口即刻开启。
